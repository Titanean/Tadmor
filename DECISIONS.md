# Tadmor — Key Decisions Log

This file is the historical record of architecture and implementation decisions, phase by phase. Standing rules and live contracts live in `CLAUDE.md`; this file is for reference when you need to understand *why* something is the way it is.

## Phase 1 (2026-03-21)

- **OkHttp over Ktor**: Switched from Ktor Client to OkHttp + direct `HttpURLConnection`-style usage. Ktor's internal buffering caused OOM on the ~6MB TAP API response. OkHttp with stream-to-file approach works reliably.
- **TLS 1.2 + HTTP/1.1 forced**: Android emulator has a TLS bug causing `DECRYPTION_FAILED_OR_BAD_RECORD_MAC` on large HTTPS responses. Fixed by forcing `TlsVersion.TLS_1_2` and `Protocol.HTTP_1_1` in OkHttp config.
- **Download-then-parse**: TAP API response is downloaded to a temp file first, then parsed from disk. This separates network I/O from JSON deserialization, avoiding both SSL corruption and OOM.
- **`largeHeap=true`**: Required because the app holds ~6,000 planet records in memory during sync and Room operations.
- **Batched Room upserts**: Stars and planets upserted in chunks of 500 to reduce memory pressure during transactions.
- **Domain module uses `kotlin.jvmToolchain(17)`** instead of `sourceCompatibility`/`targetCompatibility` to avoid Kotlin/Java JVM target mismatch.

## Phase 1.5 (2026-03-22)

- **CSV format for TAP API**: Switched from JSON to CSV format for NASA TAP API downloads. JSON responses (~6MB for planets) trigger BoringSSL `DECRYPTION_FAILED_OR_BAD_RECORD_MAC` errors. CSV is ~75% smaller (1.4MB for planets), avoiding the TLS corruption threshold. Stellar hosts (3.5MB CSV) use try-single-then-chunked-fallback with hostname-range splits.
- **3-class composition system**: Collapsed from 5 composition classes (Terra, Aquaria, Ferria, Neptune, Jupiter) to 3 (Terra, Neptune, Jupiter). Ferria/Aquaria require density data (both mass and radius), which most exoplanets lack. Even when available, the density ranges are scientifically degenerate — many planets fall in ambiguous zones. The 3-class system matches real-world exoplanet classification practice. Ferria/Aquaria visuals may be revisited later for hypothetical planet visualizations.
- **Direct sync for manual refresh**: Manual pull-to-refresh calls `repository.sync()` directly instead of through WorkManager. WorkManager's retry logic (3 attempts) swallowed errors silently. Direct calls give immediate exception access for user-visible error banners.
- **Conscrypt abandoned**: Tried Conscrypt TLS provider to fix BoringSSL issues — got "unable to parse TLS packet header". Removed entirely; CSV format workaround is sufficient.

## Phase 2 (2026-03-22)

- **Accessible UI mode**: Two curated modes — Standard (unchanged) and Accessible (larger text, bigger touch targets). Controlled via `ExoTheme.isAccessible` CompositionLocal. Typography scales ~27%, spacing ~25%. Components read the flag for structural layout changes (e.g., PlanetCard reorganizes to name/badge row + full-width stats). Icons in bottom nav stay the same size — only text labels scale.
- **WCAG text contrast**: Brightened textSecondary (#97A1B3), textTertiary (#6B7A8F), textMuted (#556275) to meet WCAG AA contrast ratios against both background and card surfaces.
- **Smallest label size bumped**: Stat labels ("MASS", "RADIUS", etc.), "Sorted by:", and expanded card detail labels promoted from `labelSmall` to `labelLarge` in both UI modes for readability.
- **Estimated SMA from Kepler's third law**: When a planet has no catalog semi-major axis but has orbital period and its star has known mass, SMA is estimated via `a = (M_star × P_years²)^(1/3)`. This feeds into temperature/insolation estimation, unlocking estimates for many more planets. Initial implementation had an extra 4π² factor inflating results ~3.4×; corrected to the simplified solar-unit form.
- **Estimated values as user setting**: Users can toggle estimated values (temperatures, spectral classes, SMA) on/off via Settings. When off, estimates are hidden from display and excluded from filter/sort operations. Toggle uses custom-drawn switch component (no Material).
- **Data availability indicator toggle**: Users can hide the "DATA" completeness dots via Settings.
- **Extended spectral class filters**: Added L, T, WD (white dwarf), and "Other" to host star filter options. `classifyStarSpectral()` handles D-subtypes (DA, DB, DC...), standard OBAFGKM+LT, Teff inference, and fallback.
- **White dwarf / exotic star colors**: `TeffColor.forStar()` uses three-tier resolution: WD override (pale blue-white regardless of Teff), Teff-based blackbody, spectral class fallback.
- **Filter sheet drag-to-dismiss**: Filter bottom sheet can be dragged down to dismiss, using `draggable` modifier with 150dp threshold. Enables peeking at results while filtering.
- **NASA hostname inconsistencies**: The NASA Exoplanet Archive sometimes lists planets in the same system under different hostnames (e.g., KOI-351 vs Kepler-90). The app treats each hostname as a separate star. No merging logic yet — noted for future work.
- **Density sanity check**: `PlanetClassifier.sanitizeMass()` nulls out mass/density when computed density exceeds 40 g/cm³ (nearly 2× osmium). Catches bad data from `pscomppars` composite table (e.g., Kepler-327 c's 102 g/cm³ from a superseded paper). Kept `pl_bmassj`/`pl_bmasse` ("best mass") columns over `pl_massj`/`pl_masse` ("true mass") because the latter drops all M·sin(i) values from RV planets, losing too much coverage.

## Phase 3 (2026-03-23)

- **GL infrastructure in `com.tadmor.app.gl`**: Eight utility classes — `ShaderSource`, `ShaderProgram`, `Mesh`, `MeshBuilder`, `CameraController`, `GLBridge`, `ExoRenderer`, `ExoGLSurfaceView`. All future 3D views build on these.
- **OpenGL ES 3.0 with VAOs**: Uses ES 3.0 Vertex Array Objects for clean buffer binding. GLSL ES 3.00 shaders stored in `assets/shaders/`, compiled and cached at first use with uniform location caching.
- **Standard vertex layout**: All meshes use interleaved position(3) + normal(3) + uv(2) at locations 0/1/2, stride 32 bytes. Consistent across all shaders.
- **Orbit camera with double-buffered matrices**: `CameraController` writes pending matrices on UI thread, atomically swaps volatile reference. GL thread reads a consistent snapshot each frame — lock-free, no synchronization overhead.
- **GLBridge via AtomicReference**: Generic `GLBridge<T>` for passing immutable parameter data classes from Compose to GL thread. Lock-free single-writer/single-reader.
- **Compose-GL integration via AndroidView**: `ExoGLSurfaceView` embedded in Compose `Box`, UI overlays (DisclaimerLabel, back button) layered on top. Touch priority: GL view consumes drag/pinch, Compose overlay elements consume their own taps.
- **Test sphere accessible from Settings → Developer → GL Test**: Validated the full pipeline. Removed in Phase 4.

## Phase 4 (2026-03-24)

- **Star map with ~4,000 host stars**: 3D point cloud using GL_POINTS with custom vertex layout: position(3) + color(3) + size(1) = 28 bytes/vertex. Not using MeshBuilder (different from standard 32-byte layout). Fragment shader creates circular points with soft edges via `gl_PointCoord`.
- **Linear distance scaling**: Star positions use linear `distPc / 30.0` (not logarithmic). Preserves relative distances and reveals structure like the concentrated Kepler search region. `PC_PER_GL_UNIT = 30` is shared between ViewModel and Renderer.
- **Star colors from Teff**: `TeffColor.forStar()` three-tier resolution (WD override → Teff blackbody → spectral class fallback). Unknown stars get dim white `(0.6, 0.6, 0.65)`.
- **Star sizes from stellar radius**: `2.5 + ln(1 + radiusSolar) * 2.0`, clamped to [2, 14]. Default radiusSolar = 1.0 when unknown. Not tied to planet count.
- **Dynamic distance rings**: Rings adapt spacing to camera zoom using a 1-2-5 interval sequence `[1, 2, 5, 10, 20, 50, 100, 200, 500, 1000, 2000, 5000]` pc. Targets ~3 visible rings. Rebuilt only when interval changes, not every frame. Rings use the star shader with `uIsLine` uniform to bypass `gl_PointCoord` discard logic.
- **Ring fog**: Rings fade to background color based on world-space distance from camera, relative to visible radius (not clip-space depth). Uses `uCameraPos` and `uVisibleRadius` uniforms. Fog scales with zoom so edge rings always fade regardless of zoom level.
- **Fixed reference screen height**: Star point sizes use a constant `REFERENCE_SCREEN_HEIGHT = 800f` rather than actual screen height. Prevents star size changes when accessible mode alters layout.
- **Sol marker at origin**: Warm yellow-white point `(1.0, 0.95, 0.8)` with size 6, always rendered at origin.
- **Ray casting for tap selection**: Linear scan of ~4,000 projected star positions per tap (<1ms). `RayCaster.pickStar()` projects stars to screen space, finds closest within threshold. `projectToScreen()` positions the tooltip.
- **Star tooltip with connector line**: Dashed line from tooltip edge to 3dp before the star. Tooltip shows hostname, spectral type (catalog or estimated), distance in user's preferred unit, planet count, and "View system →" link.
- **Estimated values use italics (not tilde prefix)**: All estimated values (spectral types, temperatures, SMA, insolation) render in `FontStyle.Italic` instead of `~` prefix. The `~` is kept as an internal data marker in format functions; rendering composables (`StatColumn`, `DetailRow`, `PropRow`, `PlanetStat`, `PropertyGrid`, `SubtitleRow`) strip it and apply italic. Setting description mentions italics.
- **System tab navigation model**: See "System Tab Page Flow & Back Behavior" section in `CLAUDE.md` for the live contract.
- **Camera pan (disabled)**: Two-finger pan infrastructure exists in `CameraController` (`panTarget`, `maxPanRadius`, midpoint tracking) but is disabled pending improvements to pan-zoom interaction. Currently two-finger = pinch zoom only.
- **Composition class colors in filter chips**: Planet class filter buttons ("Earth"/"Terrestrial", "Neptune"/"Ice Giant", "Jupiter"/"Gas Giant") use their classification colors (`ExoColors.compositionTerra/Neptune/Jupiter.text`) when inactive, matching the spectral type color pattern.
- **Center-view re-orbit**: Tooltip crosshair button calls `CameraController.lookAt()` to orbit the camera around any star. Distance rings remain at Sol (origin). Ring interval selection and fog both use `maxOf(currentDistance, distToSol)` so they scale correctly whether the camera is centered on Sol or a distant star.
- **Star map search**: Search overlay with `BasicTextField` searches all stars (unfiltered `allStars`) so filtered-out stars remain findable. Results show proper names, spectral type (colored), distance, planet count. Selecting a result uses `allPositions`/`allColors`/`allSizes` arrays, not the filtered ones.
- **Star map filters**: Four filter categories — spectral class, luminosity class, minimum planet count (slider), planet class (composition). AND between categories, OR within. Filter state is immutable (`StarMapFilterState` data class). ViewModel rebuilds filtered position/color/size arrays and posts new `StarMapParams` to bridge. Zero results correctly sets `starCount = 0`.
- **Sol filter awareness**: Sol marker visibility is gated on `isSolVisible()` which checks Sol's known properties (G2V, 8 planets, Terrestrial + Jupiter compositions) against active filters. Sol tap detection also gated on visibility.
- **Planet count slider**: Custom-drawn slider using `drawBehind` with `textMeasurer` for pixel-aligned labels at exact dot positions. Supports drag (live updates via `lastEmitted` tracking to avoid redundant callbacks) and tap-on-dot via `pointerInput(detectTapGestures)`. Step range 0 ("Any") to 8.
- **Shared luminosity class parser**: `extractLuminosityClass()` in `domain/.../classification/SpectralUtils.kt` parses spectral type strings (e.g. "G2V" → "V", "K1 III" → "III"). Checks longer patterns first (IV before I) to avoid false matches. Used by both catalog and star map filter logic.
- **Home button (return to Sol)**: Crosshair icon appears to the left of search/filter buttons when camera target is off-origin. Calls `CameraController.lookAt(0,0,0)`. Visibility polled at ~5Hz via `CameraController.isOnOrigin`.
- **Pinch-to-orbit cooldown**: 150ms cooldown after `ACTION_POINTER_UP` drops from 2→1 fingers. During cooldown, single-finger `ACTION_MOVE` updates tracking position but doesn't apply orbit rotation, preventing residual finger drag from causing unintended rapid camera spin.
- **Tab-scoped BackHandler**: All `BackHandler` calls are gated on the owning tab being the active `selectedNav`. Prevents hidden tabs (which stay composed for state preservation) from intercepting hardware back presses. `ExoSearchBar` accepts an `isActive` parameter for this purpose.
- **GL test removed**: Developer section and GL test screen removed from Settings. The test sphere served its purpose validating the Phase 3 pipeline.

## Phase 5 (2026-03-25)

- **Two-part orbital view**: System strip (Compose Canvas via `drawBehind`) embedded in star detail page + full-screen 3D orbital view (OpenGL). Avoids GLSurfaceView-in-LazyColumn compositing issues (hole-punching, z-order, clipping). The strip gives instant visual context; the 3D view gives full interactivity.
- **System strip above planet cards**: The strip sits between star properties and planet cards in the LazyColumn. It's compact (~120dp) and always visible on first load. Planet cards follow below.
- **Habitable zone from Kopparapu et al. 2013**: `HabitableZone.kt` in domain layer. Recent Venus inner edge, Early Mars outer edge. Valid for 2600–7200 K main-sequence stars. Rendered as an EvenOdd `Path` annulus (donut) in the strip, triangle strip in GL.
- **Kepler equation solver**: `OrbitalMechanics.kt` in domain layer. Newton-Raphson with 6 iterations. Used for animated planet positions in the 3D orbital view.
- **Circular orbit lines in strip**: Orbits are circles centered on the star (not ellipses). Guarantees every planet icon sits exactly on its orbit line. Estimated orbits use dashed stroke.
- **Disconnected size and distance scales**: Planet screen radii are scaled relative to each other by `radiusEarth` (largest = 14% strip height). Orbital distances are scaled independently (furthest planet at right edge). Real planet-to-orbit ratios are too small to be visible otherwise.
- **Uniform planet shrinking for overlap**: When adjacent planets would overlap, all planet icons are uniformly scaled down (preserving relative sizes) until there's 15px edge-to-edge gap. Pairs where even minimum-sized planets can't fit are skipped in the scale computation — those are handled by hiding instead.
- **Planet hiding for irresolvable overlap**: After scaling, planets that still overlap are hidden (inner planet hidden, outer kept). Orbit lines remain visible for hidden planets, signaling the user to zoom in. Greedy scan from outermost inward.
- **Horizontal zoom on system strip**: Drag left to zoom in (inner planets spread apart), drag right to zoom out. Logarithmic sensitivity. Min zoom = full system, max zoom = innermost planet at the farthest planet's default position. Zoom is a simple multiplier on distances from the star — no position recalculation needed. `clipRect` handles overflow.
- **Zoom state hoisted above LazyColumn**: `stripZoomFactor` lives in `SystemDetailContent`, not `SystemStripView`. LazyColumn recycles off-screen items, destroying `remember` state. Hoisting preserves zoom across scrolling. Passed via `zoomFactor`/`onZoomFactorChange` with `rememberUpdatedState` in `pointerInput` to avoid stale closure capture.
- **Star minimum size clamped to largest planet**: In the strip, the star circle is at least as large as the largest planet's screen circle, unless the planet genuinely approaches the star's physical radius (>30% of star radius in Earth radii). Prevents dwarf stars from appearing smaller than their planets.
- **SystemDetailContent stays composed**: Like `StarSearchContent`, `SystemDetailContent` is always composed when `systemDetail != null`, hidden with `alpha(0f).zIndex(0f)` when mode isn't DETAIL. Preserves LazyColumn scroll position and zoom state across DETAIL → PLANET → DETAIL transitions.
- **3D orbital view follows star map pattern**: Full-screen GL via `AndroidView(ExoGLSurfaceView)`, Compose overlays for controls. Top-down default camera (85° elevation). Log/linear scale toggle, play/pause, speed cycling (1x/10x/100x/1000x). Planet tap detection via `RayCaster.projectToScreen`. `OrbitalRenderer` extends `ExoRenderer`, receives `OrbitalParams` via `GLBridge`.
- **Orbital shaders**: Custom vertex layout position(3) + color(3) + size(1) = 28 bytes. Fragment shader has three render modes via `uRenderMode`: 0=point (circular + glow via `gl_PointCoord`), 1=line, 2=fill (HZ annulus).
- **Orbital view moved to Star Map tab**: The 3D orbital view lives in the Star Map tab, not the System tab. Creates a natural 3D→3D flow: star map → tap star → "View system →" → orbital view. The System tab is purely data-focused (star properties, planet cards, planet detail). "View planet →" from the orbital tooltip cross-tab navigates to the System tab's PLANET page (same pattern as Catalog's "View in System"). `StarMapViewModel` has `StarMapMode.MAP` and `StarMapMode.ORBITAL` modes. System tab no longer has an ORBITAL mode or `enteredPlanetFromOrbital` flag.
- **System strip is static**: The system strip in the System tab detail page no longer has a "View orbits" button. It's a pure 2D visualization of the system structure with no navigation action.
- **Shared buildOrbitalState**: `buildOrbitalState(detail, settings)` is a top-level function in `OrbitalState.kt`, used by both `SystemViewModel` (for the strip) and `StarMapViewModel` (for the 3D orbital view).
- **All 14 NASA limit columns**: 10 planet limits (`pl_bmassjlim`, `pl_bmasselim`, `pl_radelim`, `pl_orbsmaxlim`, `pl_orbperlim`, `pl_orbeccenlim`, `pl_orbincllim`, `pl_eqtlim`, `pl_insollim`, `pl_denslim`) + 4 stellar limits (`st_tefflim`, `st_radlim`, `st_masslim`, `st_lumlim`). Values: 0=measured, 1=upper limit (`<`), -1=lower limit (`>`). Stored through all layers (DTO → Entity → Domain). Displayed with `limitPrefix()` from `LimitFlag.kt`. Limit eccentricities and inclinations use 0.0 for orbital rendering (not reliable for animation).
- **Longitude of periapsis (ω)**: Added `pl_orblper` + `pl_orblperlim` to the data pipeline. Used in the 3D orbital view to orient each ellipse within its orbital plane. Planets without catalog ω get a deterministic pseudo-random value based on their name hash (consistent across sessions, visually varied). Displayed with limit prefix in PlanetDetailContent.
- **Longitude of ascending node (Ω)**: Not available from NASA. Always deterministic pseudo-random from planet name hash (different salt from ω). Rotates the tilt direction of each inclined orbit around the Y axis.
- **Relative orbital inclination**: Measured inclinations are relative to the system mean (subtracting the average of all measured inclinations in the system). This shows meaningful relative tilt within a system rather than absolute values (which are mostly ~90° for transit planets). Planets with limit-flagged or missing inclinations stay flat (0° relative tilt).
- **Full 3D orbital orientation**: Renderer applies three rotations per planet: ω (argument of periapsis, rotates ellipse in-plane), i (relative inclination, tilts orbital plane), Ω (longitude of ascending node, rotates tilt direction). Applied consistently in planet positions, orbit trail lines, and tap hit detection.
- **HZ extended to 2200 K**: Kopparapu et al. 2013 model parameterized for 2600–7200 K. Lower bound relaxed to 2200 K to cover ultra-cool M dwarfs (e.g., TRAPPIST-1 at 2566 K). The polynomial extrapolation remains well-behaved at these temperatures.
- **Estimated orbital period from Kepler's third law**: When a planet has SMA (catalog or estimated) but no catalog period, period is estimated via `P = √(a³/M★) × 365.25` days. Added to `PlanetClassification.estimatedOrbitalPeriodDays` and displayed in italics with `~` prefix (same pattern as other estimates). Also used in `buildOrbitalState` so planets without catalog period still animate in the orbital view at the correct speed.
- **Insolation adaptive precision**: Small insolation values (e.g., HR 8799 system) no longer display as "0 S⊕". Format adapts: ≥100 → integer, ≥1 → 1 decimal, ≥0.1 → 2 decimals, ≥0.01 → 3 decimals, else 4 decimals.
- **Planet CSV columns**: 45 columns (was 27 in Phase 1, 42 before circumbinary fields). Added `cb_flag`, `sy_snum`, `sy_name`. All `*lim` columns interleaved with their values. Database version 5 (uses `fallbackToDestructiveMigration`).
- **Circumbinary planet support**: NASA's `cb_flag` column identifies circumbinary planets. `sy_snum` gives star count, `sy_name` groups stars by system. Companion stars (e.g., "Kepler-16 B") are extracted from stellarhosts — rows sharing a `sy_name` with a primary but whose hostname isn't referenced by any planet. Stored in `StarEntity` with `isPrimary = false`. `SystemDetail` exposes `companionStars`, `combinedMassSolar`, `combinedLuminosity`.
- **Combined mass for Kepler's law**: Circumbinary planets use `M_total = M_primary + M_companion` for orbital period and SMA estimates via Kepler's generalized third law.
- **Combined luminosity for HZ**: `L_total = L_primary + L_companion` feeds into HZ calculation. Primary star's Teff drives the Kopparapu polynomial coefficients (spectral character). Luminosity estimated from `10^(logL)` or `R² × (T/T☉)⁴` when log luminosity is unavailable.
- **Binary orbital elements catalog**: `BinaryOrbitalCatalog.kt` in domain layer. Hardcoded SMA, eccentricity, and argument of periapsis (ω) for ~30 known circumbinary systems. Data from Thébault (LIRA/Observatoire de Paris) compilation and discovery papers. NASA's TAP API has no binary orbital elements table — only `cb_flag` and `sy_snum`. Looked up by `sy_name`; systems not in catalog fall back to Holman & Wiegert (1999) stability-limit estimate with circular orbits.
- **Binary star rendering (3D)**: Circumbinary systems render two animated star points on elliptical orbits around barycenter (origin), with elliptical orbit trail lines. Positions computed via Kepler equation (same `OrbitalMechanics` as planets). Each star's orbit is scaled by its mass fraction: primary at `a × M₂/(M₁+M₂)`, companion at `a × M₁/(M₁+M₂)`, always on opposite sides. Binary orbital period derived from Kepler's third law. Planet orbits remain centered at origin. Star sizes proportional to radii, colors from respective Teff values.
- **Binary star rendering (2D strip)**: Two star circles stacked vertically (above and below the planet orbit line) to stay visually separate from planets. Secondary star slightly smaller, colored by its own Teff.
- **Binary system detail layout**: For multi-star systems, the System tab detail page shows: primary star info → companion star info (same PropertyGrid style) → system strip → planet cards.
- **Companion star loading**: `ObserveSystemDetailUseCase` uses `flatMapLatest` to conditionally load companion stars when the primary has `sy_name != null` and `sy_snum >= 2`.
- **Planet CSV columns**: 47 columns (was 45 before epoch fields). Added `pl_tranmid`, `pl_orbtper` after `pl_denslim`. Database version 7 (uses `fallbackToDestructiveMigration`).
- **Real-time orbital positions via epoch data**: NASA provides `pl_tranmid` (transit midpoint BJD) and `pl_orbtper` (time of periapsis BJD) for many planets. These epoch references allow computing each planet's actual orbital phase at any given time. `OrbitalMechanics.meanAnomalyAtEpoch()` computes mean anomaly from the current Julian Date and the epoch. For transit midpoints, a π/2 phase offset approximates the transit position (near-circular orbits). Planets without epoch data fall back to the original arbitrary-phase animation.
- **Real-time reset behavior**: On initial load, the orbital view always starts with planets at their real-time positions (via `resetToRealTime()` which sets `referenceJD = currentJD` and `accumulatedDays = 0`). Moving the time slider to RT (index 0) also triggers a real-time reset. At any other time scale, positions continue forward from the current state at the selected speed. `referenceJD + accumulatedDays` gives the simulated Julian Date at any moment.
- **Time of periapsis preferred over transit midpoint**: When both `pl_orbtper` and `pl_tranmid` are available, `pl_orbtper` is used because it directly gives mean anomaly = 0, avoiding the π/2 transit approximation.
- **Sol rendered accurately**: Sol uses the same pipeline as every other star — `TeffColor.fromTeff(5778.0)` for color and `2.5 + ln(2) * 2.0 ≈ 3.89` for size — in both the star map (`buildSolMarker`) and the orbital starfield (`StarfieldData.SOL_COLOR/SOL_SIZE`). No unique hardcoded values.
- **Apparent-flux starfield dimming**: Stars in the orbital background are dimmed by apparent flux `F = L / d²`, where luminosity comes from `10^logL` (preferred), `R²·(T/5778)⁴`, or 1.0 as fallback. Brightness = `(F / F_ref)^0.3` with `F_ref = 0.01` (1 L☉ at 10 pc), floor 15%. Applied to both color and size. No normalization between stars — absolute scale. Distant Kepler systems appear sparser; nearby systems appear richer. Sol dimmed by the same pipeline.
- **Sol indicator in orbital view**: Thin circle (Compose `drawBehind`, 8dp radius, 1dp stroke) and "Sol" label (`labelSmall`) at 70% `textSecondary` opacity, projected from the starfield sphere using a rotation-only view matrix. Placed before the tooltip in the Compose Box so it renders behind tooltips and UI controls.
- **Starfield orientation from orbital inclination**: The starfield is always rotated to place Sol at a physically motivated elevation. Three cases: (a) measured inclination → Sol at `90° - i` elevation (e.g., transit at ~90° puts Sol in the orbital plane); (b) transit-discovered but no measured inclination → assumes ~90° with ±5° deterministic jitter; (c) non-transit / no inclination → deterministic random elevation 15–75° (avoids degenerate edge-on views, seeded from hostname hash with salt 5). Position angle on sky is always deterministically randomized (hostname hash, salt 3). Rotation uses Rodrigues formula around an axis perpendicular to Sol's XZ projection.
- **Slider track inset by thumb radius**: Both `PlanetCountSlider` (catalog/star map filters) and `TimeScaleSlider` (orbital view) inset the track and dots by `thumbRadius` from the Box edges. This gives overshoot room at both ends — tapping past the last dot still snaps to step 0 or max. Touch math maps `(offset.x - thumbRadius) / trackRange` where `trackRange = width - 2 * thumbRadius`.
- **ExoGLSurfaceView exclusion zones**: `exclusionZones: List<RectF>` on the GL view. On `ACTION_DOWN`, if the touch lands inside any exclusion rect, the entire gesture is passed through to Compose (`return false`). Used by the orbital view's time scale slider — its bounds are measured via `onGloballyPositioned` and published to the GL view. Prevents camera rotation from finger drags starting in the slider area.
- **Star glow compensation (CORE_FRACTION)**: The orbital fragment shader renders stars with an opaque core at 60% of point radius (`dist < 0.36`), outer 40% is transparent glow. Planets are solid to 100%. To make the visible star core match the intended size, star point sizes are inflated by `1 / CORE_FRACTION` (1/0.6) in `OrbitalRenderer`. Same compensation applied to companion stars.
- **Proportional star-planet sizing**: When a planet's radius exceeds 30% of its star's radius (e.g., brown dwarfs), the star is scaled proportionally to the largest planet rather than just clamped to match it. Below 30%, the star clamps to at least the largest planet's screen size. Applied in both `OrbitalRenderer` and `SystemStripView`.
- **Per-hostname planet count**: NASA's `sy_pnum` is system-wide, not per-star. `PlanetRepositoryImpl` overrides it with actual per-hostname count from the planet table (`planets.groupBy { it.hostname }.mapValues { it.value.size }`). Companion stars get `planetCount = 0`.
- **Circumbinary companion detection via cb_flag**: `ObserveSystemDetailUseCase` only loads companion stars when at least one planet has `cbFlag = true`, not just when `syName` matches. Prevents wide pairs (e.g., Alpha Centauri A appearing as Proxima Centauri's companion) from showing as binary systems.
- **Binary star separate tooltips**: Each star in a circumbinary system has its own tappable hitbox and tooltip. `OrbitalSelection.Star` is a `data class` with `isPrimary: Boolean`. Tap detection checks both animated binary star positions from `renderer.binaryStarPositions`. Tooltip shows the correct star's name, spectral type, and properties. Both lead to the same system page.
- **Starfield visibility toggle**: Persisted as `show_starfield` in DataStore. Button in orbital view top-right with filled 8-pointed star icon (3 stars at different sizes). Diagonal slash overlay when hidden. Gates starfield data and Sol indicator in `OrbitalParams`.
- **Habitable zone visibility toggle**: Persisted as `show_habitable_zone` in DataStore. Button in orbital view top-right (only shown when system has HZ data) with concentric circle strokes (annulus icon). Diagonal slash when hidden. Gates `hzInnerAU`/`hzOuterAU` in `OrbitalParams`. Version hash includes `showHabitableZone` to trigger GL mesh rebuild on toggle.
- **HZ annulus true alpha transparency**: Fragment shader render mode 2 outputs `alpha = 0.45` instead of 1.0. Vertex colors use dimmed terra green (`* 0.35`). Combined with standard alpha blending (`SRC_ALPHA, ONE_MINUS_SRC_ALPHA`), the starfield shows through the HZ overlay instead of being fully occluded.
- **Search bar clear buttons**: Catalog and system search bars show an X button (custom-drawn cross via `drawBehind`) when query is non-empty. Clears text and removes focus on press.
- **Orbital tooltip respects IAU proper names**: Planet tooltips in the orbital view check `ProperNames.forPlanet()` and the user's `useProperNames` setting, matching the star tooltip behavior.

## Phase 6 (2026-03-30)

- **Visual Profile Engine**: Deterministic engine in the domain layer that generates a `VisualProfile` for every planet. Pure Kotlin, no Android dependencies. Invisible to the user — drives future rendering only. The visible 3-class catalog system (Terra/Neptune/Jupiter) is unchanged.
- **Top-down bulk classification**: Rocky worlds get assigned a `BulkClass` (Ferria/Carbonia/Aquaria/Terra) representing **bulk interior** composition, then surface and atmosphere are generated within that class's constraints. BulkClass is NOT the surface — a Ferria world has an iron-dominated interior but may have a silicate crust (like Mercury). Surface composition is generated separately by `SurfaceGenerator`.
- **Composition-driven surfaces**: No rigid surface type enums. `SurfaceComposition` is percentage-based (silicates, iron, water, sulfur, carbon, nitrogen, methane, ammonia, tholins). The renderer interprets percentages + temperature/pressure to determine visual appearance (water at 90K = ice, at 300K = ocean, at 400K = vapor).
- **Bottom-up atmosphere chemistry**: `AtmosphericComposition` is molecular percentages (H2, He, N2, O2, CO2, H2O, CH4, NH3, SO2, H2S) generated from physical rules. Optical properties (Rayleigh color, scale height, clouds) derived from composition by `AtmosphereOpticsDeriver`. No rigid atmosphere type labels.
- **Atmospheric retention model**: `computeRetention()` in `AtmosphereGenerator` determines if a planet holds an atmosphere. Combines escape velocity, XUV flux (with M dwarf 1000× enhancement), magnetic field shielding, and cumulative exposure. Energy-limited escape saturation (`xuvFlux^0.6`, Lopez & Fortney 2013) prevents over-stripping. Three retention zones: <0.01 airless, 0.01–0.10 exosphere (trace gas from sputtering/outgassing), 0.10+ real atmosphere.
- **M dwarf XUV flux**: L_XUV/L_bol ≈ 10⁻³ for M dwarfs vs 10⁻⁶ for Sun. XUV luminosity computed directly relative to solar XUV, not multiplied by bolometric luminosity. Without this, M dwarf planets get ~3000× too little XUV.
- **Tidal locking timescale**: τ = C × a⁶ × M_p / (M★² × R³) where C ≈ 200 Gyr for rocky (Q~100), C ≈ 200,000 Gyr for gas giants (Q~10⁵). Derived from ω₀αQ/(3Gk₂) with moment of inertia I=0.4MR² substituted (cancels R², giving R³ not R⁵). Previous R⁵ formula was wrong by ~88,000×.
- **Greenhouse runaway**: Emergent from CO2 + H2O + insolation conditions. Venus-like amplification applied to surface pressure when CO2 > 15% and temperature is WARM or HOT. Excluded for TORRID worlds — at 2000K+ the surface is molten rock, carbonate-silicate cycle doesn't operate, atmosphere is mineral vapor (not modeled yet). Capped at 100 bar.
- **Volcanism-CO2 coupling**: Volcanic activity [0,1] feeds CO2 generation via `volcanicCO2Boost = 1 + activity × 1.5`. SO2 and H2S also scale with volcanism. Volcanism itself driven by 6-tier mass system (primary driver) plus tidal heating and age cooling (secondary).
- **H2/He on rocky worlds**: Only retained at > 8 M⊕, < 400K, retention > 0.7. Below that, XUV photoevaporation strips primordial H2/He over Gyr timescales. Significant H2/He envelopes are what define sub-Neptunes, not super-Earths.
- **Aquaria temperature gate**: BulkClassifier blocks Aquaria assignment for HOT/TORRID worlds even at low density — water has been lost, low density explained by low-iron silicate composition. WARM worlds below 1 M⊕ also excluded.
- **Tholins are emergent**: Computed in `AtmosphereGenerator.applyTholins()` when N2 + CH4 + UV flux conditions are met. Deposited on surface scaled by star age. Not assumed — calculated from atmospheric chemistry.
- **Dust Mie from surface**: Mie scattering color is a weighted blend of surface material dust colors. Strength scales with aridity and atmospheric density.
- **Deterministic from planet name hash**: `seed = planet.name.hashCode().toLong()`. Same planet always produces the same visual profile.
- **No mineral vapor atmospheres yet**: TORRID lava worlds (CoRoT-7b, 55 Cnc e) get outgassed CO2/N2 at moderate pressures (~17-36 bar) rather than the rock vapor envelopes (Na, K, SiO, Fe at ~2000-4000 atm) that SpaceEngine models. Mineral vapor is a future enhancement.
- **No O2 implying biology**: Abiotic O2 from H2O photolysis only, kept at trace levels.
- **AtmosphereOptics matches shader parameter model**: `AtmosphereOptics` data class mirrors the atmosphere.html proof-of-concept shader uniforms exactly. Scale heights, cloud altitudes, and atmosphere thickness in km (no normalization to planet radius, no clamping). Density as a multiplier relative to Earth (1.0 = Earth surface density ≈ 1.225 kg/m³). Per-channel RGB scattering/absorption coefficients for Rayleigh, Mie, and absorption bands. Cloud texture parameters (size, distortion, bumpiness, banding). Fog with scale height and patchiness. Sun intensity from stellar luminosity / distance².
- **Universal Rayleigh base coefficients**: All atmospheres use the same base Rayleigh R/G/B (5.5 / 13.0 / 33.0 from Earth preset). No per-composition cross-section scaling — the density multiplier handles absolute scattering strength. Per-composition tuning deferred to the rendering phase where visual results can be evaluated. The λ⁻⁴ wavelength ratio is always preserved.
- **Mie per-channel scatter + absorption**: Mie scattering and absorption are separate per-channel RGB vectors derived from surface dust composition. Iron oxide dust absorbs blue/green (appears red). Tholins → orange-brown (Titan-like). Silicate → neutral. Carbon → dark broadband absorber. Volcanic ash adds to both scatter and absorption.
- **Absorption band for ozone / CH₄ / H₂S**: The shader's ozone layer is repurposed as a general altitude-dependent absorption band. O₂-bearing atmospheres get actual ozone (Chappuis band). CH₄-rich atmospheres (ice giants, cold worlds) get methane red/NIR absorption → blue appearance. H₂S-rich atmospheres get broad blue absorption.
- **Volcanic activity uses exponential cooling**: Mass-dependent geological half-life model: `halfLife = mass × 10 Gyr`. Moon-sized bodies dead in <1 Gyr, Earth active at 4.6 Gyr, super-Earths essentially never cool. Tidal heating scaled by √mass. Hot world bonus gated by mass > 0.5 M⊕.
- **Tidally locked polar caps reinterpreted by renderer**: `polarCapExtent` stores the amount of ice coverage regardless of tidal lock state. The renderer checks `tidallyLocked` to decide placement — geographic poles for rotating worlds, anti-stellar hemisphere for locked worlds.

## Phase 7a (2026-04-01)

- **Single-pass full-screen quad ray marcher**: Direct port of `atmosphere.html` to GLSL ES 3.0. No separate geometry pass — the fragment shader does ray-sphere intersection, scattering integration, surface shading, clouds, fog, tone mapping, and sun disk in one pass. `PlanetRenderer` extends `ExoRenderer`, draws `MeshBuilder.quad()` with depth test and backface culling disabled.
- **Reinhard luminance-preserving tonemapper**: Replaced ACES with `maxC / (1 + maxC)` applied to the max channel, scaling all channels proportionally. ACES compressed cool star tints toward white at high intensities. Reinhard preserves hue ratios across the full brightness range.
- **Auto-exposure from sun intensity**: `exposure = 20 / sunIntensity`, derived from the geometric mean of atmosphere.html preset exposure/intensity pairs. No atmospheric effects factored in — purely stellar flux-based. Frigid planets no longer appear black, torrid planets no longer blow out.
- **Blackbody star colors with saturation boost**: Sun color uses Tanner Helland blackbody approximation (`TeffColor.fromTeff()`) with a 1.2x power boost on non-dominant channel ratios. Replaced the pale step-function `ColorPalettes.starTint()`. TRAPPIST-1 (2566K) now renders as deep orange ≈ (1.0, 0.58, 0.24), close to the atmosphere.html preset target #ff8943.
- **Cloud noise baking**: Cloud FBM is pre-rendered to a 4096×2048 equirectangular R16F texture via a separate bake shader (`cloud_bake.frag`). Runs once per planet (when seed changes). The main shader samples cloud density, normals, and shadow opacity from cheap texture lookups instead of computing 5-octave FBM per ray step. Cloud normals use texture gradient approximation (3 offset samples, `uCloudBumpiness` scales the gradient).
- **Reduced ray march for mobile**: 16 primary steps (iSteps), 4 shadow steps (jSteps), 3-octave FBM (reduced from 64/8/5 in the web PoC). Fog patchiness and Mie dirtiness sampled once at the ray midpoint, not per step. Early transmittance termination (`r+g+b < 0.03`) breaks the loop when the view ray is fully occluded.
- **Airless early-out**: When `uAtmosphereThickness <= 0.0`, the shader skips the entire ray march — just ray-sphere intersection + Lambertian diffuse. Covers ~20% of planets.
- **GL view always fullscreen**: The `AndroidView` wrapping `ExoGLSurfaceView` is always `fillMaxSize()`. Mode transitions (HALF ↔ FULL) are handled by camera target offset and Compose content overlay, not by resizing the GL surface. This eliminates the visible warping caused by `onSurfaceChanged` aspect ratio changes during transitions.
- **Camera roll for HALF mode**: `CameraController` gained a `roll` property with Rodrigues' rotation applied to the up vector in `rebuildMatrices()`. In HALF mode the camera faces the sun (fully lit hemisphere) with 90° roll so gas giant bands appear vertical. The target is offset in the screen-down world direction (`+Z` for the rolled view) to push the planet upward, revealing only the top portion above the text content.
- **FULL mode 75% sunlit**: Camera positioned 60° off the sun direction (`sunAz - 60°`, elevation 15°, roll 0°). The `cos(60°) = 0.5` relationship gives `(1 + cos(θ))/2 = 75%` visible lit fraction. The terminator is visible near one edge, providing depth cues without hiding the planet in shadow.
- **Content starts at fixed 200dp**: `GLOBE_HALF_HEIGHT = 200.dp` is the hard-coded spacer before text content begins, placing the content start roughly at the planet's half-radius point. Camera offset (`0.9 * radius`) scales with planet size in the GL view, maintaining approximately consistent visual proportion across different planets.
- **Flat surface color for all planets**: `SurfaceColorBlender` computes a single RGB from `VisualProfile` — weighted blend of `ColorPalettes` material colors for rocky worlds, average of band colors for gas giants. Phase 7b adds textures (terrain heightmaps, gas giant fluid simulation).
- **Static deterministic clouds**: `uTime` is set to a fixed value derived from `profile.seed` (not animated). Each planet always shows the same cloud configuration. Cloud animation deferred to a later phase.
- **km units in shader**: Planet radius, atmosphere thickness, scale heights, and cloud altitudes all in km. Float32 precision is adequate at orbital camera distances (1.5–8× planet radius). No normalization needed.
- **Sun direction from axial tilt + season phase**: `sunDirX/Y/Z` computed from `profile.axialTilt` and a deterministic season phase (seeded from profile). Base azimuth is fixed at 90° (lit from the right). Each planet shows a unique illumination angle based on its axial tilt and where it is in its "year."
- **No haze layer**: Removed from the atmosphere.html port since `AtmosphereOptics` has no haze fields. Fog with scale height and patchiness provides sufficient limb effects.
- **Preset debug screen**: `AtmospherePresetScreen` in Settings loads hardcoded Solar System presets (Earth, Mars, Venus, Jupiter, Titan, Neptune, Triton, TRAPPIST-1e) with exact values from atmosphere.html. Provides pixel-comparable reference for shader port verification and gas giant atmosphere tuning.
- **ExoRenderer background pure black**: Changed `glClearColor` from `#06080F` to `(0,0,0,1)`. Atmospheric scattering needs true black to look correct; the old color was imperceptibly close anyway.
- **Per-composition Rayleigh cross-sections**: Rayleigh scattering base coefficients (5.5/13.0/33.0) are scaled by a composition-weighted refractive index ratio relative to Earth's N₂/O₂ mix. CO₂-heavy atmospheres scatter ~2.4× more (cross-section ∝ (n−1)²), H₂/He atmospheres scatter ~0.2×. Prevents all planets from having Earth-like blue limbs regardless of composition.
- **Fog pressure scaling**: Fog density multiplied by `√(surfacePressure)` (clamped 0–1 bar). Prevents thin-atmosphere worlds (Mars-like) from having Earth-level fog. Only dense atmospheres (Venus, Titan) produce substantial fog.
- **Venus cloud gate removed**: Originally required H₂O > threshold for Venus-like clouds. Hot CO₂ worlds with desiccated atmospheres (h2o < 0.05) now get thick sulfuric acid clouds unconditionally when CO₂ > 50% and pressure > 10 bar. Separate dry greenhouse path at >700K handles Venus analogs that lost water early.
- **Color tinting for variety**: `tintColor()` applies per-channel random shifts (±8 to ±20 per type) to ARGB Long cloud/fog colors. Combined with `applySpice()` post-generation pass (±10–20% on scattering, absorption, density, cloud parameters), ensures no two planets look identical even with similar compositions.
- **CH₄ absorption band fixed width**: Ice giant methane absorption uses fixed center (10–20 km) and width (12–20 km) instead of scale-height-proportional values. Large scale heights (20–60 km for ice giants) were producing absorption widths of 14–72 km, causing over-absorption and too-green appearances vs Neptune preset's 16 km.
- **AMMONIA chromophore absorption ordering**: Mie absorption channels ordered B > G > R (blue/UV absorbed most) matching real sulfur/phosphorus chromophore physics. The orange/brown appearance of Jupiter-class planets comes from surface cloud deck color, not from differential limb absorption.
- **Super-puff atmosphere extent**: Atmosphere thickness cap raised from 3,000 → 20,000 km, Rayleigh scale height cap from 200 → 4,000 km. Super-puffs (Kepler-51 c/d: ~0.03 g/cm³) have true scale heights of thousands of km due to extremely low gravity. The old caps truncated the atmosphere at ~46% of surface density, creating an artificial hard edge visible as a colored limb ring. Normal gas giants (Jupiter H≈27 km, Neptune H≈20 km) are unaffected — their values were well below the old caps.
- **Hycean worlds deferred to Phase 7b**: Sub-Neptunes like K2-18 b already look correct as gas giants in the atmosphere renderer. The visual distinction (ocean surface) requires surface textures, which is a 7b deliverable. No special atmosphere handling needed.

## Phase 7b (2026-04-03)

- **GPU bake to equirectangular RGBA8 texture**: Gas giant surfaces are baked to a 1024×512 FBO once per planet (on seed change), same pattern as `cloud_bake.frag`. Bound to `GL_TEXTURE0` as `uPlanetTexture`; `uUseTexture = 1` flips the existing sampling path in `planet.frag`. Near-instant on device (~1 frame).
- **Perlin noise via permutation texture**: 256×1 R8 texture with NEAREST filtering. CPU generates a Fisher-Yates shuffled permutation from `permSeed`, uploaded via `glTexSubImage2D`. `texelFetch` gives exact integer lookup in GLSL. Different seed = different permutation = different planet.
- **Fluid advection (2 steps)**: Curl noise velocity + domain warp + macro/micro storm vortex fields + zonal wind rotation. Reduced from PoC's 3 steps for mobile performance. ~80 Perlin evaluations/pixel total.
- **sRGB round-trip**: Bake shader outputs `pow(color, 1/2.2)`; `planet.frag` decodes with `pow(tex, 2.2)`. Ensures correct linear-space lighting in the atmosphere ray marcher.
- **Storm generation on CPU**: Macro (1–3 GRS-type) and micro storms (pearl strings + cyclone clumps) generated in `buildGasGiantBakeData()` using `DeterministicRandom(seed xor 0x47415347)`, passed as `vec4` uniform arrays. Max 3 macro, 20 micro storms.
- **THOLIN gas giant type**: Added for cold/frigid Neptune-class planets with photochemical methane haze. Purple/violet palette over H₂/CH₄ atmosphere. Probabilistic assignment: ~30% of FRIGID, ~20% of COLD Neptunes become THOLIN; the rest remain ICE_GIANT (blue). Previously all cold Neptunes were forced to THOLIN due to a fixed methane proxy check.
- **Multiple palette variants per type**: Each gas giant type has 3–5 named palette variants (e.g. AMMONIA: Red Jupiter, Pale Saturn, Dark Belt, Green Phosphine, Ochre Blaze). Generator picks one randomly per planet, ensuring planets of the same type look distinct. Previously all planets of a type shared a single palette.
- **`bandCount` independent from `bandingStrength`**: Added `bandCount: Float` to `GasGiantProfile` as a separate parameter. Previously band count was derived from `bandingStrength` via `2 + bandingStrength × 8`, conflating two independent visual properties.
- **HSV per-band color variation**: Each physical band gets a unique hue rotation via HSV conversion in GLSL (`shiftHue` / `scaleSaturation` functions). Magnitude matches the PoC: `perlin3D * 0.15 * contrast` (≈±54° at full contrast). Replaced a buggy RGB channel shift (`+R, -0.3G, -0.7B`) that caused pink/lime artifacts on achromatic ice giant palettes.
- **Band width variation via phase distortion**: Low-frequency Perlin noise sampled at ~1/3 the band frequency acts as a non-uniform phase offset before `sin()`, compressing some bands and widening others. Strength scales with `(1 - uBandSoftness * 0.5) * 0.75` — sharper bands get more irregular widths.
- **Breakup/count tuned by type**: Neptune types (ICE_GIANT, SUB_NEPTUNE, THOLIN) get higher `bandBreakup` (0.45–0.80) and lower `bandCount` (1.5–4) than Jupiter types (AMMONIA: 0.20–0.50 breakup, 4–7 bands). Prevents ice giants from showing too many regular bands.
- **Contrasting pole colors**: 35% of gas giants get a pole color from a type-specific palette (POLE_JUPITER_BLUE, POLE_SATURN_GREEN, POLE_ICE_WHITE, POLE_AURORA_TEAL) instead of a desaturated band color.
- **Oblateness formula corrected**: The `(5/4)ω²R³/GM` uniform-density formula overestimates by ~1.7× for centrally-concentrated gas giants. Scaled by 0.6 to reproduce Jupiter's actual oblateness (0.065). Minimum rotation period raised from 4h → 8h (no real giant spins faster than Jupiter's ~9.9h). Cap lowered from 0.15 → 0.10 (Saturn-level maximum). Eliminates the excess of egg-shaped planets.
- **noiseScale per type**: SUB_NEPTUNE = 2.5, ICE_GIANT/THOLIN = 3.0, all others = 4.0. Lower frequency for smaller planets avoids the miniature-Jupiter appearance.

## Phase 7c (2026-04-05)

- **Shared terrestrial/gas giant bake FBO**: Only one planet renders at a time, so `gasGiantFBO`/`gasGiantTexture` were renamed `surfaceFBO`/`surfaceTexture` and reused for terrestrial bakes. Same 1024×512 RGBA8 allocation serves both surface pipelines. Terrestrial bake uses a separate `terrestrialBakeShader` but identical FBO + permutation texture infrastructure.
- **Height packed in alpha channel**: Terrestrial bake outputs `vec4(sRGB_color, rawHeight)`. `planet.frag` samples the alpha at three offsets to compute a gradient → perturbed tangent-space normal for bump-mapped Lambertian shading. Gas giants upload `uBumpStrength = 0` (bands are flat); terrestrials upload `roughness × 0.8`.
- **Bump direction (tangent space)**: The perturbation is `T * (hC - hU) + B * (hC - hV)` — normal points *away* from rising terrain. Initial implementation had this inverted (`hU - hC`), which cast shadows backwards and made craters look like hills. Fixed.
- **Independent terrain vs crater bump scales**: `TERRAIN_BUMP_SCALE = 0.25` and `CRATER_BUMP_SCALE = 0.32` are separate GLSL constants. Terrain stays subtle (rolling hills don't cast dramatic shadows) while craters read as crisp depressions.
- **Sea level as a true height threshold**: `uSeaLevel ∈ [0,1]` is the elevation cutoff below which a pixel is ocean. Below threshold → ocean color + **flat heightmap**: `heightOut = 0.5 + (uSeaLevel - 0.5) * TERRAIN_BUMP_SCALE + (roughTex - 0.5) * 0.008`. Oceans are genuinely flat in the bump map (no terrain shadows leaking through blue water) with tiny capillary ripples. Crater bowls below sea level automatically fill as circular bays.
- **Scaled water sea level**: Raw water fraction is remapped via `(raw × 1.65).coerceIn(0, 0.94)` so water worlds actually get ocean-dominated surfaces instead of continents with puddles. Capped at 0.94 to always preserve island specks.
- **Composition prominence curve**: Trace species (carbon, sulfur, tholins) contribute disproportionately to material palettes via `prominence(frac) = sqrt(frac)`. At 10% carbon the palette weight is ~31%, not 10%. Reflects that minor but visually dominant components (graphite veins, sulfur frost) should be seen even in small quantities.
- **Two-tier cratering**: 30 CPU-generated explicit craters (power-law size distribution, `radius = maxCraterScale × uniform^sizeExponent`, degradation from `CraterProfile`) + Worley cellular background noise for dense small-crater fields. Worley is skipped when `craterProfile.density < 0.1` (geologically active worlds like Io or Earth).
- **Monotonic red→orange→yellow lava gradient**: Lava oceans use a three-stop thermal palette with strict monotonic ordering — `lavaCool (0.55, 0.020, 0.005)` deep red → `lavaMid (1.00, 0.26, 0.015)` bright orange → `lavaHot (1.00, 0.78, 0.08)` saturated yellow. Heat coordinate from signed FBM signal (warmth + flow + cracks octaves), biased toward cool end (0.38 midpoint) so yellow is rare and marks only the hottest channels. Colors never reverse — always cool→hot in wavelength.
- **Extra-saturated lava palette**: Channel floors pushed hard toward pure hues (B < 0.08 on yellow, B < 0.02 on orange) because TORRID worlds have extreme auto-exposure → low `uCameraExposure` → Reinhard compression pulls brighter colors toward white. Near-grey palettes desaturate; saturated hues survive.
- **Lava emissive mask by low B**: `planet.frag` detects warm lava pixels via `smoothstep(0.40, 0.75, R) * smoothstep(0.15, 0.05, B)`. Keyed off the blue channel directly because lava's red→orange→yellow gradient sits entirely below B=0.10 (deepest channel: lavaCool B=0.005, lavaHot B=0.08) while warm terrain (silicate tan, iron rust, volcanic ash) always carries B > 0.15. The previous `R-B margin` test leaked: tan/rust have R-B of 0.25–0.30, overlapping the threshold and making bright silicate terrain falsely glow under 9× additive emission. Gated behind `uLavaEmission` uniform so only worlds with actual lava (>900K surface OR volcanism >0.70) get the emissive treatment.
- **Blackbody thermal emission uniform**: `uThermalEmission` is a CPU-computed RGB from `computeThermalEmission(surfaceTempK)` using Tanner-Helland blackbody with intensity `((T - 700) / 2300)^2.6`. Added to surface color as `uThermalEmission * baseSurfaceColor * 4.0` on both airless and atmosphere paths. Hot rock glows independently of sunlight; modulated by surface albedo so dark basalt radiates less than light silicate.
- **Tidally-locked polar cap baselines**: Eyeball worlds get elevated ice coverage since their entire anti-stellar hemisphere is permanently cooling. Baselines by temperature class: FRIGID 0.75, COLD 0.62, TEMPERATE 0.50, WARM 0.40. The 50% TEMPERATE baseline ensures the eyeball pattern is always visible (effect emerges at ~50% coverage). Rotators use the original climate-based polar cap calculation unchanged.
- **HotRockyOutcome weighted selector**: Hot rocky worlds (TORRID/HOT Terras) no longer default to Venus-analogue 9/10 times. `rollHotRockyOutcome()` chooses between VENUS_THICK (base 60%), THIN_CO2 (25%), and STRIPPED_RESIDUAL (15%), with weights shifted by star type (M-dwarfs strip atmospheres harder), age, and water fraction. Matches real sample: 55 Cnc e / CoRoT-7 b are Venus-thick; TRAPPIST-1 b / LHS 3844 b are stripped; GJ 1132 b is thin.
- **Silicate cloud decks for lava worlds (>1500K)**: Day-side vaporized Na/K/SiO/Mg/Fe circulates to cooler regions and condenses as MgSiO₃ + FeO particulates (Kite et al. 2016). Kicks in instead of the cloudless dry-greenhouse path. Low altitude (8–25 km), partial wind-driven coverage (45–75%), strong banding. Alkali flame tint (Na/K emission) layered on top of the base ash palette.
- **Volcanic-ash silicate palette**: Silicate cloud colors are warm ash-brown (R > G > B by 18–38 per entry, e.g. `0x3A2C22` to `0x523C2C`) not neutral slate-grey. Neutral palettes compress toward white through Reinhard tonemapping under TORRID sunlight; warm-biased ash preserves hue character through brightness compression.
- **Differentiated rotation rates by bulk class**: Jupiter-class giants get 10h mean / 8h minimum rotation (Jupiter ≈ 9.9h, Saturn ≈ 10.7h). Ice Giants get 16h / 12h (Uranus ≈ 17.2h, Neptune ≈ 16.1h). Sub-Neptunes get 26h / 14h — they formed with less angular momentum and are more susceptible to tidal spin-down. Previously all giants used one rotation rate formula.
- **Piecewise cloud coverage remapping**: Cloud coverage was too hidden at < 0.4 threshold, so water worlds appeared cloudless. `promoteCloudCoverage()` applies a 4-piece curve: raw 0.0→0.35 maps roughly linearly (0–0.35, wispy), 0.35–0.50 remaps steeply to 0.40–0.65 (the visible-cloud onset zone is expanded), 0.50–0.80 stays 0.65–0.85 (cloudy baseline), 0.80–1.00 stays 0.85–1.00 (global cover, rare). Makes "partly cloudy" worlds actually appear cloudy without making everything globally overcast.
- **Volcanic hotspots (terrain-coupled)**: Dry worlds with volcanism > 0.70 get sparse lava patches painted directly into the surface texture. Distribution noise runs at `scale * 3.0` (fine enough for numerous small patches, still below base-terrain's finest octaves). Patch boundaries follow the heightmap via `hotspotNoise + (0.5 - elev) * 0.30` — lowlands boost toward the threshold, highlands suppress below it — so patches trace terrain contours instead of floating as independent blobs. Heat coordinate is scaled by `mix(0.75, 1.15, 1 - elev)` so deep basins burn yellow-hot while ridges run deep red. Volcanic activity controls **coverage** (threshold shifts from 0.90 at vol=0.70 to 0.72 at vol=0.95) while opacity stays binary — partial edge pixels would still trigger lavaMask emission on the night side, so `heat *= hotMask` collapses edges to lavaCool. Gated `uSeaLevel <= 0.0`: water worlds would quench exposed magma instantly.
- **Water specular glint**: Ocean pixels get Blinn-Phong specular with Schlick Fresnel. Detected in `planet.frag` by the water-like base color signature (low R, moderate G, high B) and modulated by `uWaterSpecular` from the profile. Only active on worlds with surface temperature 273–373 K and real sea coverage. Gives water oceans a sun glint off the terminator-facing hemisphere; dry, frozen, and lava oceans get no specular.
- **Smooth shoreline blend**: Sea↔terrain transition replaced the old `bool isOcean = elev < uSeaLevel` hard cutoff with a `seaMask = smoothstep(seaLevel + 0.012, seaLevel - 0.012, elev)` coverage mask. Land palette is sampled unconditionally (submerged pixels clamp to palette index 0 for a beach colour), then sea colour and its lava gradient crossfade in via `seaMask`. Both the colour mix and the heightmap flatten follow the same curve. Matches the blend style of the volcanic hotspot painter — no more nearest-neighbour waterlines.
- **GL view visibility gate per tab**: `AndroidView(GLSurfaceView)` draws to a native window-layer Surface that Compose's `alpha(0f)` and `zIndex(0f)` cannot cover. Before this fix, switching from the planet page (globe visible) to the Star Map tab would leave the globe's surface punching through and block the star map. Each GL view (`PlanetGlobeView`, `StarMapScreen`, `OrbitalScreen`) now carries `update = { glView.visibility = if (tabIsActive) VISIBLE else GONE }`. GLSurfaceView tears down its surface on GONE and recreates it on VISIBLE — state is preserved, rendering stops on the inactive tab.

## Phase 7d (2026-04-06)

- **Cloud/fog drift animation**: Cloud layer rotates around the Y axis at a fixed rate (`uCloudDriftSpeed = 0.04 rad/s`, ~2.5 min per revolution). Implemented as a Y-axis rotation matrix applied to the sampling direction in `sampleCloudNoise()` before equirectangular projection. `uTime` is elapsed seconds since planet load (reset on seed change). Fog patchiness also uses `uTime` for subtle drift. Rate is higher than the PoC's `0.02` to compensate for the missing cloud shape morphing (PoC re-bakes FBM every frame with domain warp).
- **Planetary ring rendering (single-pass in planet.frag)**: Rings rendered as an equatorial Y=0 plane intersection in the existing full-screen ray marcher. No separate geometry pass. Ten ring uniforms: `uHasRings`, `uRingInner/Outer` (planet radii), `uRingOpacity`, `uRingColors[5]`, `uRingColorCount`, `uRingGapCount`, `uRingDustiness`, `uRingSeed`. Ring intersection computed once before the airless/atmosphere branch; both paths share the results.
- **Ring density from multi-frequency noise**: `getRingDensity(r)` combines four noise scales: coarse envelope (15×), fine bands (40×), very fine detail (80×), and micro ringlets (200×). Coarse and fine are additive; the two high-frequency octaves are signed perturbations that carve into and boost the coarse structure, creating sharp visible micro-ringlets and gaps within broader bands. Deterministic gaps carved via `hash3`-positioned Cassini-like divisions (up to 4). Edge fade at inner/outer boundaries. `dustiness` controls the density threshold: low = solid Saturn-style bands, high = thin Uranus-style ringlets.
- **Ring color with micro variation**: `getRingColor(r)` interpolates between up to 5 colors across radial position, then applies ±37% brightness variation from two high-frequency noise octaves (80× and 200×) matching the density micro detail. Creates visible bright/dark ringlet structure even where density stays relatively uniform.
- **Bidirectional ring shadows**: Ring shadow on planet: `getRingShadow()` traces sun ray from surface point to Y=0 plane, samples ring density at intersection → darkens surface proportional to ring density (dense sections cast full shadow, thin sections cast partial). Planet shadow on ring: `ringInPlanetShadow()` does ray-sphere intersection from ring point toward sun → pure black shadow (no ambient).
- **Ring lighting model**: Flat Lambertian with `abs(sunDir.y)` — rings scatter light from both top and bottom faces. Minimum 5% ambient prevents fully black edge-on rings.
- **Ring atmosphere attenuation**: Rings behind the planet (visible around the atmospheric limb) are injected as the background color *before* atmosphere back-layer attenuation, so the full atmosphere optical depth properly dims the ring through dense limbs. Rings in front of the planet (between camera and surface) are composited after atmosphere processing. All blending in linear space before tonemapping.
- **Differentiated ring profiles by composition class**: Jupiter-class gets wide Saturn-like systems (0.6–2.0 radii span, high opacity, low dustiness 0.05–0.35, 1–4 gaps). Neptune-class gets narrow Uranus-like rings (0.2–0.8 radii span, lower opacity, high dustiness 0.5–0.95, 0–1 gaps). Terra gets modest debris rings. 20% crossover chance: Jupiters can get Uranus-style thin rings and Neptunes can get Saturn-style solid rings.
- **Ring breakup gaps**: 25% chance of an extra gap on any ring. Shader gap widths range from 2% to 18% of the annulus (power-curved toward narrower), allowing dramatic visual splits.
- **Random ring width modifier**: Annulus width scaled to 10–100% of base value per planet (`uniform^0.6`, median ~63%). Produces everything from vestigial ringlets to full systems. Applied before Hill sphere clamping.
- **Hill sphere ring constraint**: `RingGenerator` computes the planet's Hill radius in planet radii from SMA, planet mass, and star mass. Rings must fit within 1/3 of the Hill radius (Domingos et al. 2006 stability limit). Close-in planets (R_Hill < 5 planet radii) lose 30% ring probability; marginal cases (R_Hill < 10) lose 15%. Ring outer radius is also physically clamped to `R_Hill / 3`. Hot Jupiters at 0.05 AU effectively can't get rings.
- **Mass-scaled terrestrial ring probability**: Base 8% for Terras, scaled by mass: sub-Earths (<0.5 M⊕) halved, super-Earths get up to +8% (>5 M⊕). Heavier rocky worlds have stronger gravity wells to capture and retain ring debris.
- **RingProfile from visual profile engine**: `RingGenerator` in domain layer assigns rings probabilistically (Jupiter 30%, Neptune 15%, Terra 8%), with temperature/age/mass/Hill sphere modifiers. Radii 1.2–3.6 planet radii (clamped by Hill sphere). Low-saturation icy vs rocky color palettes from `ColorPalettes`. `PlanetGlobeParams` maps `RingProfile` fields directly to shader uniforms. Deterministic seed from planet name hash XOR'd with `0x52494E47` ("RING").

## Phase 8 (2026-04-12)

- **2D planet icons from VisualProfile**: `IconProfileBuilder` reads the real `VisualProfile` (not approximations) so icons match the 3D globe. `PlanetIconProfile` carries body/accent colors, atmosphere tint, cloud coverage+density, polar cap extent, crater count, ring presence, and land threshold — all derived from engine output.
- **Terrain as 96×96 value-noise bitmap**: Two-octave value noise at low frequencies (0.09/0.22) for large continental shapes. Binary threshold separates body from accent material. Upscaled with `FilterQuality.None` for a chunky stylized look. Cached per `(seed, bulkClass)` in `remember`.
- **Cloud noise bitmap**: Separate 96×96 noise texture with smoothstep threshold driven by `cloudCoverage` and per-pixel alpha scaled by `cloudDensity`. Wispy clouds (TRAPPIST-1 f) render at ~35% opacity; thick decks (Venus, Earth tropics) reach near full. No limb fade — clouds render flat. Cached per `(seed, cloudCoverage, cloudDensity)`.
- **Volatiles excluded from terrain materials**: Water, nitrogen, methane, and ammonia are never included in `dominantSolidColors`. On the globe these only appear below `uSeaLevel` (as ocean/ice) or as polar caps — never as terrain above sea level. Terrain materials are strictly rock-type: silicates, iron, sulfur, carbon, tholins.
- **Iron oxide requires real O₂ atmosphere**: `canOxidizeIron` gates on `atm.present && o2 > 0.05 && surfacePressureBar > 0.10`. Prevents rust-red accents on airless/thin-atmosphere worlds (TRAPPIST-1 e/h).
- **Water above 373 K excluded as surface material**: Vapor is not a visible solid — eliminates white-painted hot dry worlds (L 98-59 d).
- **Sea-level display requires substantial atmosphere (≥0.50 bar)**: Liquid water and ice seas on the icon require `surfacePressureBar >= 0.50`. Exospheric and thin-atmosphere worlds render barren even if the engine assigns nonzero seaLevel. Sea-dominated threshold raised to seaLevel ≥ 0.55; solid-with-seas threshold raised to ≥ 0.30.
- **Lava world gate at 900 K**: Matches the globe's Phase 7c lava-emission threshold. Below 900 K the surface is hot rock but not visibly molten. Engine seaLevel is zeroed for 373–900 K worlds (`temperaturePermitsSea` filter).
- **Volcanic hotspot patches**: `volcanicActivity > 0.70 && temp >= 300 && profile.seaLevel < 0.05` — matches globe's Phase 7c threshold. Suppressed when engine intended any kind of sea (TOI-7166 b water+lava overlap fix). TRAPPIST-1 b/c qualify; Teegarden/YZ Cet cool tidal worlds don't.
- **Atmosphere as tinted disk overlay**: Semi-transparent circle at `1.10 × diskRadius` drawn over terrain (replaces the old radial-gradient halo outside the disk). Alpha driven by log-density: `(0.08 + densityFactor × 0.70)` clamped 0.08–0.78. Venus ≈ 0.74, Earth ≈ 0.32, Mars ≈ 0.08. Gated on `hasRealAtmosphere` (≥ 0.10 bar).
- **Fixed-distance rings**: Icon rings at 1.18R–1.42R with 0.24R stroke regardless of engine ratios. Prevents overflow into text content at small icon sizes.
- **Crescent polar caps for tidally locked worlds**: `Path.op(diskPath, dayPath, Difference)` carves a night-side lune. Crescent width = `polarCapExtent × 0.32 × diskRadius`, clamped 6–28% of radius. Only shown when `polarCapExtent >= 0.40`.
- **Oval polar caps for rotators**: `drawOval` with `capRx = (extent × 1.10 + 0.15) × diskRadius` and `capRy = 0.60 × capRx` for a flattened ice-sheet appearance, positioned at 78% of disk radius toward the top.
- **15% minimum accent fraction**: `dominantSolidColors` requires the second material to be ≥ 15% of the surface. Below that, accent falls back to a darkened body color. Eliminates spurious trace-mineral accents (5% iron becoming 50% of the icon). Accent fraction amplified ×1.25 and clamped 0.30–0.55 so patterns are visible at small sizes.
- **Per-planet dominant color**: `PlanetIconProfile.dominantColor` (interpolation of body + accent) used for per-planet colors in the system strip and 3D orbital view, replacing the old composition-class-based coloring.
- **Raised water sea levels**: Aquaria TEMPERATE/WARM 0.40–0.75 (was 0.30–0.70), COOL 0.20–0.55 (was 0.10–0.40), FRIGID 0.10–0.35 (was 0.05–0.25). Terra water floor raised from 0.05 to 0.15, ceiling from 0.35 to 0.45. Globe `scaleWaterSeaLevel` (×1.65, cap 0.94) amplifies these further.
- **Surface view deferred indefinitely**: SPEC.md Phase 8 originally called for a surface-level camera with high-resolution terrain, sky dome Rayleigh scattering, host star at correct angular diameter, and day/night cycle. Dropped primarily due to performance constraints — the planet globe ray marcher already pushes mobile GPU budgets with 16 primary + 4 shadow ray march steps, cloud bake, and terrestrial/gas giant surface bakes. A surface view would require a second full-screen ray marcher (sky scattering) plus high-resolution terrain mesh generation, roughly doubling GPU load on a screen that would only be visited briefly. Phase 8 was repurposed for 2D planet icon refinement instead.

## Phase 9 — Polish (2026-04-12)

- **Globe renderer pause on scroll**: When planet detail content scrolls past the globe area (`scrollState.value >= GLOBE_HALF_HEIGHT`), the GL view switches to `RENDERMODE_WHEN_DIRTY`, stopping the draw loop. Scrolling back up resumes `RENDERMODE_CONTINUOUSLY`. GL context and textures survive — no re-bake needed.
- **Planet size comparison section**: "COMPARE" section at the bottom of the planet detail page. Shows the exoplanet icon beside a Solar System analogue (Earth, Neptune, or Jupiter) scaled by real radius ratio. The larger planet fills 80dp; the smaller is proportionally scaled with no minimum floor (accuracy over legibility — Earth beside Jupiter is ~7dp). Both icons sit in fixed-height boxes with `Alignment.Center` so the visual midpoint stays stable when the dominant planet swaps on chevron cycling.
- **Solar System icon profiles**: `SolarSystemProfiles.kt` in the domain layer. Hardcoded `PlanetIconProfile` constants for Earth (ocean blue body, green land accent, 40% cloud coverage, polar caps), Neptune (4 subtle blue bands, Great Dark Spot), and Jupiter (7 accurate bands: cream/white/orange/white/orange/white/cream, Great Red Spot). `SolarPlanet` enum carries the profile, real radius in Earth radii, and display name. `defaultFor(CompositionClass)` selects the analogue matching the exoplanet's class.
- **Spot overlays on planet icons**: `PlanetIconProfile` gained `spotColor`, `spotY`, `spotWidthFrac`, `spotHeightFrac` fields (default 0 = no spot). Rendered as a clipped oval slightly right of center after bands but before the atmosphere overlay. Used for Jupiter's Great Red Spot and Neptune's Great Dark Spot.
- **Chevron cycling order**: Up chevron cycles Earth → Neptune → Jupiter → Earth (increasing size). Down chevron cycles the reverse. Chevrons are custom-drawn V-strokes (1.2dp, `textSecondary`) aligned with the Solar System column only, not centered.
- **Bilinear terrain interpolation**: Terrain bitmaps switched from `FilterQuality.None` (nearest-neighbor, chunky pixel art) to `FilterQuality.Low` (bilinear) for smoother icon rendering. Cloud bitmaps already used bilinear.
- **Star globe on star detail page**: Mirrors the planet globe pattern. `StarGlobeView` composable wraps an `ExoGLSurfaceView` with `StarRenderer` (extends `ExoRenderer`), fed by `GLBridge<StarGlobeParams>`. Camera: roll 0°, HALF mode with `lookAt(0, -radius * 0.9, 0)` pushing the star upward so text content sits below. HALF mode explicitly faces the equator (azimuth/elevation both 0°) and keeps roll at 0° so gas-giant-baked brown dwarf bands render horizontally rather than pole-to-pole. Full-screen quad ray marcher (`star_globe.frag`, `planet.vert`), no separate geometry pass.
- **Procedural star surface**: Single-pass ray-sphere intersect + limb darkening + multi-layer convection noise. Surface noise combines four components: supergranulation (low-freq `surfPt * 2.2`), domain-warped FBM granulation (3-octave, drives primary cell structure), fine detail (`uNoiseScale2`), and ridged filaments on a stretched coordinate space (`1 - |noise|` raised to power 3) suggesting chromospheric fibrils. Hot/cool tint contrast is wide (`uStarColor * 1.5` vs `uStarColor * (0.45, 0.35, 0.25)`) so bright cells stand out against dark intergranular lanes. Emissive mask `smoothstep(-0.35, 0.35, convection)` adds `hotTint * intensity * 2.0` to bright upwelling regions, analogous to the lava emissive path in `planet.frag`.
- **Radial corona (no filaments)**: The original ShaderToy-style polar-coordinate filamentary corona was dropped — it didn't scale with zoom (wisps grew to fill the screen) and read as noise rather than glow. Replaced with a hard-clamped two-term radial falloff: tight inner halo `exp(-(r-1)*8)` + soft outer `pow(1 - tOuter, 3)` where `tOuter = clamp((r-1)/coronaExtent, 0, 1)`. Strictly vanishes at `(1 + coronaExtent)` radii — no infinite exponential tail leaking past the camera frame. Distance computed in km (world space) so falloff scales correctly with camera zoom.
- **Per-star seed offset in procedural noise**: Every procedural star was sampling `noise3d(surfPt * scale + vec3(0, uTime*k, 0))` — the same unit-sphere coordinates for every star, so the underlying lattice pattern was identical across all stars. Fixed with a `uNoiseSeedOffset` vec3 uniform added to every noise sample (supergran, domain warp, FBM, fine detail, ridged, BD shimmer). Renderer derives the offset from `params.seed` by slicing the 64-bit hostname hash into three 16-bit chunks mapped to −500..+500 so nearby hostname hashes land far apart in noise space.
- **Star animation rates bumped ~10×**: Original `uTime` multipliers (0.008–0.035) against noise coordinates of magnitude 20–80 produced ~0.05%/sec drift — effectively static. New rates (0.08–0.85) give readable motion: supergranulation drifts over ~15 s, FBM granulation churns over ~3 s, fine detail shimmers at ~1 s. Ridged filaments also get a small time term so chromospheric streaks flow.
- **Gentle convection cell radius scaling**: Convection cell size scales with pressure scale height `H_p ∝ T·R²/M`; the cell-to-disk ratio `H_p/R ∝ R/M`, so giants show a few enormous cells (Betelgeuse-style) and dwarfs show fine granulation. First implementation had the scaling too aggressive, making red giants look like trivially blobby clouds. Final formula: `radiusScale = (1/radiusSolar^0.35).coerceIn(0.6, 1.6)`, applied as `noiseScale1 = 40 * radiusScale`, `noiseScale2 = 80 * radiusScale`. Gentle curve + tight clamp keeps both ends plausible.
- **Reinhard-max tonemap preserves star hue at all brightnesses**: Because the scene uses luminance-preserving Reinhard on the max channel (not per-channel, not ACES), a red M dwarf stays its tinted color no matter how bright the HDR input — G and B asymptote at their ratio of R rather than climbing toward 1.0 together. SpaceEngine-style "bright star disc bleeds to white" does not happen here. This is the Phase 7a decision in action; spectral type stays legible as a visual cue at any zoom level. If SpaceEngine-like saturation is ever wanted, the fix is localized: either per-channel Reinhard or a luminance-driven desaturation gated in `bloom_composite.frag` on a per-renderer uniform, without touching the planet path.
- **Brown dwarf procedural/bake split at L/T boundary**: L dwarfs (≥1300 K, class `L`) render through the normal procedural star path — their photospheres are hot enough (~1500–2400 K) to read plausibly as deep red stars. T dwarfs and cooler (or Teff < 1300 K) render via the existing gas giant bake pipeline for banded clouds. Detection: `firstClass in listOf('T', 'Y') || teff < 1300`. White dwarfs are filtered out first (`D` + subtype letter).
- **Brown dwarf bands were pole-to-pole (bug fix)**: Original BD equirectangular sampling divided `n.y` by `uStarRadius` (~695700 km), which collapsed the `v` coordinate to a single texture row and produced visible pole-to-pole banding. Fixed by using the normalized sphere normal directly: `ny = clamp(n.y, -1, 1); v = (asin(ny) + π/2) / π; u = 1 - (atan(n.z, n.x) + π)/(2π)` — matching `planet.frag`'s mapping exactly. Camera roll/HALF framing then delivers horizontal bands as expected.
- **Brown dwarf two-colour palette**: `BD_T_VARIANTS` in `ColorPalettes` uses a strict alternation of deep purple and salmon-red with three variant hues (`#2A0E3D/#FF6678`, `#1F0830/#FF7588`, `#381650/#E85569`). Band count 8–13 (more than Jupiter; dwarfs have deeper atmospheres with more observable strata). Emissive glow applied to the salmon highlights: luminance-driven mask `smoothstep(0.18, 0.50, lum)` multiplies the texture color by `intensity * 0.9` so bright bands pop past the tonemapper. Previous `BD_WARM/BD_COOL/BD_TRANSITION` palettes removed.
- **Spectral-class Teff fallback on star page**: Original `buildStarGlobeParams` defaulted `teff ?: 5778` and called `TeffColor.fromTeff(teff)` directly, so stars without measured Teff (e.g. DENIS-P J082303.1-491201, an L dwarf) rendered with the Sun's color. Fixed by (a) looking up a spectral-class default Teff (`O 35000`, `B 15000`, ..., `L 1900`, `T 1000`, `Y 500`) for parameter tuning, and (b) resolving color via `TeffColor.forStar(star.teffK, star.spectralType)` which has the same three-tier WD-override → blackbody → spectral-class fallback used by the system strip and orbital view. Star color is now consistent across all three surfaces.
- **HDR bloom post-process on planet + star globes**: `BloomPipeline` helper in `com.tadmor.app.gl` owns an HDR FBO (`RGBA16F`, with `RGBA8` fallback if `GL_EXT_color_buffer_half_float` is missing), a half-res bright-extract FBO, and a half-res blur FBO. Three shaders in `assets/shaders/`: `bloom_extract.frag`, `bloom_blur.frag` (9-tap separable Gaussian), `bloom_composite.frag`. Pipeline: scene shader renders linear HDR to the HDR FBO with `uPostprocess = 1` → bright extract (half res) → separable blur (2 H/V iterations for a wider glow) → composite tonemaps HDR once and adds LDR bloom additively before sRGB encode. Applied to `PlanetRenderer` and `StarRenderer` only; star map and orbital view stay on the direct path.
- **LDR-space bloom extraction**: First version of bloom extracted in *linear HDR* space — after exposure, ordinary lit pixels on a planet are ~15–20, so any reasonable threshold caught the entire disc and the blurred result just lifted everything uniformly (invisible glow). Fix: `bloom_extract.frag` tonemaps once internally (matching Reinhard) and thresholds in LDR [0, 1] space. Hot regions concentrate near 0.9–1.0 post-tonemap and cleanly exceed the threshold; cool regions stay dim. `bloom_composite.frag` also tonemaps the HDR scene once and adds the LDR bloom additively — no double tonemap. Final settings: planet `threshold=0.55, intensity=0.8, blurRadius=3.5`; star `threshold=0.55, intensity=1.6, blurRadius=5.0, softKnee=0.75`.
- **Scene shaders gate tonemap behind `uPostprocess`**: Both `planet.frag` and `star_globe.frag` got a `uniform int uPostprocess`. When set to 1 (bloom path), the fragment shader writes `col * uCameraExposure` as linear HDR — the composite pass handles tonemap + sRGB. When 0 (default), the old direct path runs (`tonemap → pow(1/2.2)`). `planet.frag` needed this at all four output sites (airless hit, airless miss, no-atmosphere sky, main atmosphere path), consolidated into a single `writeFinal(vec3 col)` helper next to the `tonemap()` function to keep the four call sites terse.
- **Tab and page slide transitions**: Swiping between tabs (CATALOG/SYSTEM/STAR_MAP/SETTINGS) and between System-tab pages (SEARCH/DETAIL/PLANET) uses an iOS-style push convention — new content enters from the direction of travel, old content exits opposite. Direction is derived from enum `ordinal` (higher index → enter from right). Tabs stay composed with `key(NavDestination.X) { Box(modifier = tabModifier(X)) { ... } }` inside a single weighted `Box`. `tabModifier` reads progress in a `graphicsLayer` lambda (draw phase) so per-frame updates don't recompose. 280ms duration, `FastOutSlowInEasing`.
- **ExoBottomNav hoisted to MainActivity**: A single `ExoBottomNav` lives in `MainActivity` below the tab `Box`, not inside each screen. Keeps the nav bar static during slide transitions. `SystemScreen` requests the nav be hidden (globe fullscreen mode) via an `onHideBottomNavChange: (Boolean) -> Unit` callback; MainActivity tracks `systemHidesNav` and only respects it when `selectedNav == SYSTEM`.
- **Synchronous progress writes beat `Animatable + LaunchedEffect`**: The `Animatable` + `LaunchedEffect { snapTo(0f); animateTo(1f) }` pattern causes a one-frame flash on transition start: the recomposition triggered by the selection change runs before the effect, so `graphicsLayer` reads the old end-state (`progress = 1f`) and snaps the new content into its final position for one frame before the effect resets progress. Replaced with `var progress by remember { mutableFloatStateOf(1f) }` written synchronously during composition (`if (selection != previous) { progress = 0f }`, guarded so it doesn't infinite-loop), driven forward by a `LaunchedEffect(previous) { withFrameNanos loop with FastOutSlowInEasing.transform }`. First post-change draw reads `progress = 0f` — no flash. Applied identically in `MainActivity` (tabs) and `SystemScreen` (pages).
- **GL surfaces stay visible during tab transition via `isTabOnScreen`**: `GLSurfaceView` draws to a window-layer Surface that Compose `alpha(0f)` / `zIndex` cannot cover — the Phase 7c fix was to toggle `glView.visibility = GONE` when leaving the tab. That flipped instantly on `selectedNav` change, unloading the surface mid-transition (visible "star map disappears then slides off"). Fix: `MainActivity` computes `isStarMapOnScreen` / `isSystemOnScreen` via `derivedStateOf { selectedNav == X || (tabProgress < 1f && fromNav == X) }` — true while the tab is animating out as well as when selected. Threaded into `StarMapScreen`, `SystemScreen`, and `OrbitalScreen` as `isTabOnScreen` for both `glView.visibility` and `viewModel.setVisible()`. `derivedStateOf` caps recomposition to animation start/end boundaries, not every frame. `BackHandler` gating stays on the strict `selectedNav == X` check to prevent two tabs' handlers from competing mid-transition.
- **Single persistent `CameraController` per globe view**: `PlanetGlobeView` and `StarGlobeView` originally used `remember(params.seed) { CameraController(...) }` so each planet/star got a fresh controller. Bug: `AndroidView(factory = { ... })` captures `cameraController` in a closure and runs the factory exactly once — the `PlanetRenderer` / `StarRenderer` on the GL thread holds that original reference forever. On planet navigation the fresh controller was dangling; the renderer kept using the stale one with the previous planet's `minDistance`/`maxDistance` bounds, target offset, and roll. Manifested as wrong camera distance (old bounds clamped `setDistance` to the wrong range), off-center rotation (old `target = (0, 0, radius_prev * 0.9)` with a new-radius planet), and swapped drag axes (stale roll from a previous rolled view). Fix: one `remember { CameraController() }` for the composable's lifetime, never re-created. A second `remember(params.seed, radius, sunAz, globeMode) { /* update fields in place */ }` block runs synchronously during composition to rewrite bounds, distance, orbit angles, roll, and target on the same controller the renderer already references.
- **Camera config must run synchronously during composition, not in a `LaunchedEffect`**: `LaunchedEffect(globeMode, sunAz) { apply HALF/FULL state }` ran **after** composition committed, so the GL thread could render its first frame(s) reading the `CameraController`'s constructor-default state (distance=3f, target=origin) before the effect fired — visible as a one-frame burst of wrong camera positioning on planet page init. Consolidated into a single `remember(keys) { update fields }` block that runs during composition before `AndroidView`'s factory (which runs in layout, strictly after commit). The very first GL frame sees fully configured matrices.
- **Shared refresh-spin pattern (RegenerateButton + PullToRefresh)**: Both use a manual `Animatable` driven by `LaunchedEffect(isRefreshing)` rather than `rememberInfiniteTransition`, so spin-up starts at 0° (no phase jump from whatever the persistent transition drifted to) and stop can decelerate smoothly. Shared `SPIN_PERIOD_MS = 800` keeps the two icons visually continuous when the pull indicator hands off to the button. `RegenerateButton` decelerates to the next full rotation (`ceil(rotation.value / 360f) * 360f`, `LinearOutSlowInEasing`, 400ms) then snaps to 0; `PullToRefresh` keeps the arrow at 0° static offset during pull and grows the arc sweep in place, so the arrow isn't whipping around while the tail draws in.
- **PullToRefresh `onPreFling` must not block the fling**: Original implementation used `coroutineScope { launch; launch }` inside the suspending override, which stalled the list fling ~300–400ms on every release while the offset animation played — the whole list appeared to freeze before scrolling. Fix: launch reconciliation animations on the outer `rememberCoroutineScope` and return `Velocity.Zero` immediately. Early-return when `offsetAnim.value <= 0f || isRefreshing` so normal list scrolls (no overscroll) never kick the animator at all.

Dismiss × custom-drawn with 1.2dp stroke + 0.3 pad ratio for consistent visual weight.
- **"Sorted by:" label size + animated toggle**: Label bumped `labelLarge` → `bodyMedium` in both UI modes for readability. Arrow is a canonical down-pointing silhouette wrapped in `rotate(Animatable)`; targets are `0°` (descending) and `-180°` (ascending) so toggling down→up interpolates through negative angles (visually CCW) and up→down back to 0 interpolates positive (visually CW). 260ms tween with `FastOutSlowInEasing`. - **Smooth camera pan on star map recenter**: Tooltip crosshair and Sol home button no longer call `cameraController.lookAt()` directly — they go through a local `animatePanTo(x, y, z)` helper that drives the target via `withFrameNanos` + `FastOutSlowInEasing.transform` over 420ms. Size-scaled (16dp standard / 20dp accessible) with 1.2dp stroke matching the icon library convention.
- **Globe HALF↔FULL camera animation**: `PlanetGlobeView` and `StarGlobeView` interpolate camera orbit angle, elevation, roll, target offset, and distance over 420ms (`FastOutSlowInEasing`) — slightly slower than the 280/240ms UI crossfade so the camera lingers through the mode switch rather than snapping. Planets rotate through (sunAz, 0°, roll 90°, target +Z 0.9R) ↔ (sunAz−60°, 15°, roll 0°, centered); stars translate (0°, target −Y 0.9R) ↔ (15°, centered) with azimuth held at whatever the user rotated to. `CameraController.azimuth` / `elevation` / `roll` now have public getters.
- **Start-state capture must be synchronous, not in `LaunchedEffect`**: First attempt captured `startDistance = cameraController.currentDistance` inside `LaunchedEffect(globeMode)`, but Compose runs effects *after* composition/SideEffect commits. The first post-flip `SideEffect` saw stale start values (still the previous mode's captured state) and snapped the camera to formula defaults; only then did `LaunchedEffect` fire and capture the now-already-snapped state. Visible as: zoom "snaps to default, then animates from default." Fix: a `lastMode` sentinel compared synchronously during composition (`if (globeMode != lastMode) { capture; lastMode = globeMode }`) writes the start-state `mutableFloatStateOf`s *before* the SideEffect in the same composition reads them. Same pattern as Phase 7c's synchronous camera-config insight, applied to animation driver state.
- **Shortest-path azimuth wrap on mode return**: `CameraController.azimuth` accumulates unbounded from touch drag (no modulo), so a user who rotates the planet 350° in FULL and swipes back to HALF would see the camera spin the long way back over 420ms if lerped linearly. The SideEffect wraps `(endAz − startAz) mod 360` into `[−180, 180]` so the return always takes the shorter arc. Only needed for planet's azimuth (roll is 0↔90 with known endpoints; elevation is clamped ±89°; stars hold azimuth).
- **Ring-aware max zoom-out**: `PlanetGlobeView`'s `maxDistance = max(8R, ringOuter × 6 × R)`. The ×6 is derived from *horizontal* FOV, not vertical: `perspectiveM` takes 45° vertical FOV, and on a ~0.5 portrait aspect that gives ~23° horizontal FOV. - **Sun disc over-exposure: whole-disc near-white desaturation**: The luminance-preserving Reinhard tonemap (max-channel Reinhard) preserves color ratios exactly, so a TRAPPIST-1-like orange sun (`uSunColor ≈ (1, 0.6, 0.3)`) post-tonemap was a flat strongly-orange disc — spectrally accurate but perceptually wrong, since real eyes/cameras saturate the bright disc to near-white when the source exceeds sensor dynamic range. - **Star halo and flare scale with apparent flux, not exposure-compensated**: `uSunIntensity` already encodes `L / d²` (apparent flux at the planet), and `uCameraExposure ≈ 20 / uSunIntensity` is its inverse, so anything multiplied by `starLight × exposure` ends up exposure-compensated to a constant — fine for the disc (we want it visible at any distance) but wrong for the halo/flare, which physically should fade with distance just like a real star looks like a faint pinprick from Pluto, not a haloed sun. Introduced `haloAtten = clamp(uSunIntensity / 40, 0, 1)` (40 ≈ Sun-at-1AU reference) and multiplied glow + opticalFlare contributions by it, defeating the auto-exposure cancellation. - **Procedural diffraction spikes (general approach)**: Per-pixel `computeStarSpikes(rayDir, angle, haloAtten)` overlays an irregular eye/lens-style starburst on every fragment. Built from a sum of `pow(|sin(k·θ + φ)|, p)` terms at irregular frequencies and phases — varied k gives different spike counts per term, varied p gives different ray thickness, varied φ prevents alignment. θ is computed by projecting `rayDir` onto a basis perpendicular to `uSunDirection` and taking `atan2`. The radial profile is `(1 − angle / maxAngle)²` clamped to non-negative. Reads as an irregular messy starburst (not the clean N-spike pattern of a camera) — matches the messy bursts seen with imperfect lens/cornea optics rather than a clean aperture diffraction. Reference image: `AstroNiki.png` in repo root.
- **Diffraction spike compositing — overlay everything (refactored `writeFinal`)**: Spikes are an optical/lens artifact, not a sky element, so they must overlay the entire scene including the planet surface. `writeFinal` was refactored to take `rayDir` and call `computeStarSpikes` internally before exposure/tonemap, centralising the contribution. All 4 callers (airless surface hit, airless miss, atmosphere-miss bgFinal, full atmosphere path) now pass `rayDir`. The two prior in-place spike additions in the disc blocks were removed. Net: any pixel — even one rendering planet surface — gets the spike contribution added on top, exactly mirroring how camera lens flare doesn't care about scene depth. The existing `computeSunOcclusion` still kills spikes when the sun is fully behind the planet (no source light = no flare), but for visible suns the spike pattern correctly overlays the planet silhouette.
- **Spike brightness envelope is a peaked function of apparent flux**: The spike intensity is *not* monotonic in apparent brightness — it peaks at moderate flux and fades on both ends. Bright-end fade: `spikeStrength` includes a `dimness = 1 − haloAtten` factor so close bright suns drown out spikes (lens artifacts disappear when the source overwhelms the sensor). Dim-end fade: `dimFade = smoothstep(0.005, 0.10, uSunIntensity)` — sub-threshold flux carries no signal to bloom, so very distant planets like Pluto-equivalents see their spikes fade out too. Combined effect: bright Sun-at-1 AU and ultra-dim Pluto-distance both get faint spikes, with the peak around 1–10 AU mid-range. Length and intensity have separate envelopes (`lengthMod` 0.72 close → 0.82 far; `intensityMod` 0.15 close → 0.60 far) so close stars give short tight whisper-spikes while far stars give long pronounced rays. Per-ray length modulation (`perRayLength = 0.40 + 0.60 × Snorm` where `Snorm = S(θ) / 3`) means brighter pencil cores extend further while dim halo wisps terminate sooner — natural ray-length variation within a single starburst.
- **Spike occlusion: brightness drops faster than length**: Both planet-occlusion and screen-edge-occlusion fold into a single `sunOcc` scalar that affects spikes in two ways: (1) `maxAngle *= sunOcc` (length shrinks linearly), and (2) brightness multiplies by `pow(sunOcc, 2.5)` (steeper-than-linear drop). At `sunOcc = 0.5`, length is at 50% but brightness is at ~17% — rays visibly fade *away* during partial cover instead of just shrinking uniformly. Planet occlusion via `computeSunOcclusion()`: `smoothstep(0.90, 1.10, ratio)` over the closest-approach distance from the camera→sun line to the planet center, so spikes stay near full strength while any part of the disc is visible (`ratio = 1.0` = sun center on the limb, ~50% covered) and only drop after full cover.
- **Spike screen-edge fade (camera frame as additional occluder)**: Without this, the still-on-screen portion of a starburst whose disc has moved off-screen would emanate from "nowhere" — the disc is gone but spike pixels nearer the centre of the screen still see the sun direction. `computeScreenEdgeFade` projects `uSunDirection` to NDC by extracting the camera basis from `uInverseView`'s columns (`camRight = uInverseView[0].xyz`, etc.) and reading the FOV scaling off `uInverseProjection`'s diagonal. `dEdge = max(|ndcX|, |ndcY|)` is `< 1` inside the screen and > 1 outside; `1 − smoothstep(0.90, 1.10, dEdge)` gives a fade that completes near the actual frame edge. Multiplied into `sunOcc` so it shrinks spike length and dims spike brightness the same way planet occlusion does.
- **Hazy circular halo at spike convergence point (constant size, gentler occlusion fade)**: When the planet covers the sun and spikes shorten via `maxAngle × sunOcc`, they collapse toward an infinitesimal point — visually weird. Added a `pow(1 − angle / 0.025, 2) × 0.6` constant-radius (~1.4°) hazy circle to the spike contribution. Crucially, the haze does *not* multiply by `pow(sunOcc, 2.5)` like the spikes — only by linear `sunOcc` — so during partial occlusion it stays prominent while the spikes fade fast around it, giving the convergence point a soft glow body rather than a vanishing pixel. Both haze and spike feed into the same `total` value going to the white-saturation mix, so they bloom together visually. Once `sunOcc → 0` (fully covered), both go to zero.
- **Spike noise function tuned for heavy blur (final)**: After several rounds of tuning between razor-thin pencils and over-blurred uniform glow, settled on heavy blur with subtle directional structure. `pow` exponents: cores at 3–8 (down from initial 50–130 in the crisp first pass), halos at 1.0–1.8 (near-uniform fill). Total summed weight ~5.0 across 9 noise terms; normalised to ~3.0 for `Snorm`. Reads as a soft hazy bloom with embedded ray-like brightness variation rather than discrete lines — closer to the eye/cornea reference (AstroNiki.png) than to camera diffraction. The tuning iterations are illustrative: first attempt used pow 50–130 (crisp), then user requested doubly blurrier (halved to 9–32), then doubly blurrier again (halved to 5–16), then once more (halved to 3–8). Each halving roughly doubles the angular footprint of each ray.
- **Spike base brightness multiplier and dim-end clamp**: Final spike contribution is `total × spikeColor × uSunIntensity × 0.10` — the `0.10` keeps base post-tonemap brightness modest (peak post-Reinhard ≈ 0.3 for close stars, ≈ 0.7 for far stars) so spikes are visible but never blanket the sky. White-blowout for high-amplitude pixels is `mix(uSunColor, vec3(1.0), clamp(total × 0.45, 0, 0.65))` — capped at 65% white so dim/distant suns don't go full bleach, while bright peaks still saturate convincingly.
- **Spike envelope re-tuned against 55 Cancri reference system (final)**: After settling on a peaked-with-distance envelope (entry above), iterative testing against the 55 Cnc planets — Janssen at 0.015 AU and Harriot at 0.78 AU around a 0.61 L☉ host both saturate `haloAtten` at 1.0 (`dimness = 0`); Lipperhey at 5.74 AU lands at `uSunIntensity ≈ 0.74`, `dimness ≈ 0.98` — showed close/medium were perfect at the prior values but far was too long and too bright. Final formulas: `intensityMod = 0.15 + dimness × 0.05` (0.15 close → 0.20 far) and `lengthMod = 0.72 − dimness × 0.45` (0.72 close → 0.27 far). Both far-end values are ~⅓ of where they previously sat (0.59 → 0.20, 0.82 → 0.27), while close/medium are unchanged. Length now decreases monotonically with distance, which physically models "less light reaching the camera = less to bloom" rather than the prior "exposure boost = more dramatic spikes" model. The earlier monotonically-increasing length model was overshooting at typical exoplanet distances (a few AU); the catalog has many more Lipperhey-equivalents than Pluto-equivalents, so the bias should favour those.
- **Settings page section order (DISPLAY → CONTROLS → UNITS → UNIT SYMBOLS → NAMING → DATA)**: Ordering reflects access frequency and scope of impact, not alphabetical or implementation history. DISPLAY first (UI size + render-on/off toggles like neutron-star rotation — the first thing a new user might toggle for accessibility/preference). CONTROLS second (camera input — also a first-touch decision once the user starts navigating 3D views). UNITS / UNIT SYMBOLS together in the middle (data-format preferences — touched once and rarely revisited). NAMING follows (semantic preferences — Earth vs Terrestrial, Neptune vs Ice Giant — even less frequently changed). DATA last (estimated values, data availability indicator — power-user concerns that change appearance of every screen).
- **Neutron star rotation toggle**: New `UserSettings.neutronStarRotation: Boolean = true` field controls whether pulsars/neutron stars visibly rotate in the star globe view. Persisted in DataStore as `KEY_NEUTRON_STAR_ROTATION`. Threaded through `buildStarGlobeParams(star, neutronStarRotation = true)` which gates `pulsarSpinRate = if (neutronStarRotation) 100f else 0f` — single-point gate at param construction time, not in the renderer, so a setting change recomputes the params and the renderer just sees `spinRate = 0`. `SystemScreen` keys its `remember(starHostname, settings.neutronStarRotation)` block on the setting so toggling it recomputes params without remounting the surface view. Default on (preserves the dramatic ~16-rotation/sec look that's a signature of the pulsar visualisation); off serves motion-sensitive viewers and clean reference shots.
- **Spectral-class filter renamed `WD` → `D`, added `Q` for pulsars**: Display label normalised to the actual MK convention — white-dwarf spectral types start with `D` (DA, DB, DC, etc.), so the previous `WD` shorthand was an internal-only abbreviation showing through in the UI. Spectral filter order in both `FilterBottomSheet` (catalog) and `StarMapFilterSheet` (star map) is now `O B A F G K M L T Q D` (catalog adds trailing `Other`). New `Q` slot uses the synthetic pulsar/neutron-star class introduced in #350; `TeffColor.fromSpectralClass("Q")` returns the saturated electric blue (`#7AAEFF`). Classifiers updated: `CatalogViewModel.classifyStarSpectral` checks `star.isPulsar()` first (returns `"Q"`), then white-dwarf D-prefix detection now returns `"D"` instead of `"WD"`. `StarMapViewModel.matchesFilter` adds an `if (star.isPulsar()) "Q"` short-circuit before falling through to `spectralClassLetter` / `estimateSpectralFromTeff`. `resolveSpectralType` and `spectralClassLetter` both renamed their white-dwarf return value to `"D"`. `TeffColor.fromSpectralClass` keeps both `"WD"` and `"D"` as accepted inputs so the internal `forStar` lookup (which still passes `"WD"`) stays compatible — only user-facing labels switched. No DataStore migration needed because filter selections aren't persisted.
- **Surface specular auto-detected per pixel (water glint + ice highlights + rock gloss)**: Replaced the prior water-only `pow(nDotH, 220) × Schlick-fresnel × 3.0` specular block with a `surfaceSpecular(...)` helper that auto-detects material from `baseSurfaceColor` and applies a per-material profile in a single call. **Water** (`B − R > 0.05`): `pow(nDotH, 80)` × Fresnel `0.10 + 0.90 × pow(1−nDotV, 4)` × 8 — wider than the prior pencil-sharp 220 exponent, with a real head-on Fresnel base (was 0.02, now 0.10) so sun glint reads from any viewing angle, not only at the limb. **Ice** (`min(R,G,B) > 0.45` AND not water): `pow(nDotH, 50)` × softer Fresnel × 6 — broader, brighter head-on (ice is more diffuse than mirror water). **Rock** (lit, not water/ice/lava-dark): `pow(nDotH, 12)` × 0.10, no Fresnel — broad weak sheen captures damp stone / mineral facets without claiming a mirror finish. The `uWaterSpecular` uniform was repurposed (kept name to avoid Kotlin-side renames) as a global on/off gate × strength scaler; Kotlin-side now sets `waterSpecular = 1f` for every terrestrial bake (was 0.85 only for liquid-water worlds, 0 otherwise) so ice worlds and dry rocky worlds also get the right specular automatically. Gas giants take a different params path that never sets `waterSpecular`, so it falls through to 0 — no spurious gloss on Jupiter/Neptune textures from the per-pixel detection. Helper called from BOTH the airless surface path and the atmosphere surface path (previously only atmosphere had any specular at all).
- **Liquid water bake is flat; ice seas keep depth-derived topology**: In `terrestrial_bake.frag`, `seaColor = uColorWater * (1 − depth × 0.35)` produced visible noise-driven splotches inside ocean basins (the same fractal noise that drives terrain elevation also drove water "depth", reading as fake bathymetry from orbit). Real ocean is essentially uniform from space, so removed the depth multiplier for liquid water. Frozen seas (TRAPPIST-1 f-class) however ARE solid surfaces with geological time for cracks, ridges, refrozen leads, and pressure-ridge topology — so a new `uniform bool uWaterIsIce` gates `waterDepthAmount = uWaterIsIce ? 0.25 : 0.0`. Ice seas keep ~25% of the depth darkening to suggest ice topology; liquid stays flat. Plumbed through `TerrestrialBakeData.waterIsIce: Boolean`, set to `true` only in the `temp < 273f` branch of the water-colour selector. Lava (own depth-driven thermal gradient), hydrocarbon, and hot-desert branches all leave it `false`. Renderer uploads as `glUniform1i(0|1)` since GLSL `bool` accepts integer.
- **Diffraction spike occlusion smoothstep range scales with disc angular diameter**: Original fixed range like `(0.80, 0.99)` for `computeSunOcclusion` and `(0.90, 1.10)` for `computeScreenEdgeFade` was wrong for both ends of the catalog: close-orbiting Janssen-class worlds (sun's ~17° disc fills a huge arc) had spikes vanishing while the disc was still mostly visible, while distant systems (pinprick disc) had spikes lingering well after the disc had passed behind the limb. **Planet occlusion**: derived `sunInRatioUnits = (sunAngularRad × cameraDist) / uPlanetRadius` — converts the disc's angular radius to the same units as the closest-approach `ratio`, then `smoothstep(1 − sunInRatioUnits, 1 + sunInRatioUnits, ratio)` adapts per scene. For Janssen this gives smoothstep range ~(0.1, 1.9) (huge); for Pluto-distance ~(0.9995, 1.0005) (razor-tight, snap at the limb). **Screen edge**: `sunNdcRadius = sunAngularRad / min(invProj[0][0], invProj[1][1])` converts to NDC half-radius; smoothstep range becomes `(1 − sunNdcRadius, 1 + sunNdcRadius)`. Disc-aware on both ends — close discs keep spikes alive until the disc is mostly past the frame, distant pinpricks transition sharply right at the screen edge.
- **Ring opacity uses extinction-style power curve (mix HDR fix)**: Linear `mix(col, ringLit, ringDens)` produced visually broken compositing when `col` carried HDR-bright contributions (especially the star disc, exposure-boosted glow, optical flare). Even at `ringDens = 0.7`, attenuating an HDR value of ~50 by 30% still gave ~15 post-mix, which Reinhard saturated to ~0.94 — read as the disc bleeding through any moderate ring density, "as if rings were always behind the star". The math was correct front-to-back compositing in linear HDR; the issue was the perceptual non-linearity of tonemap on bright sources. Centralised the fix in a `ringOpacity(d) = 1 − pow(1 − d, 2.5)` helper applied at all 5 mix sites (airless surface hit, airless miss, atm-miss bgFinal, ring-behind-atm, ring-in-front-of-atm). Sharpens the falloff: 0.1 → 0.23, 0.4 → 0.72, 0.7 → 0.95, 0.9 → 0.997 — thin sections stay translucent, dense sections (Saturn B-ring optical depth ≈ 1.5) properly occlude bright backgrounds. As a side benefit the ring decks read as more solid in normal viewing too, which had also been a low-grade complaint about the prior look.
- **WISPS ring tier (faintest class, mostly terras)**: New fourth `RingStyle` below `DEBRIS` for ultra-faint dust torus — opacity 0.02–0.10, dustiness 0.85–1.0 (no banded structure), no gaps, wider radial extent (0.4–1.2 vs DEBRIS's 0.2–0.5) modelling dust spread under radiation pressure. Style selection re-weighted: terras now roll **70% WISPS / 30% DEBRIS** (most Earth-class rings read as a barely-perceptible dust torus, the more visible debris ring is the rarer outcome — physically realistic since impact-debris dust is the only mechanism a terra-mass world has to retain rings); giants additionally get a 5% chance of WISPS each (Jupiter-Halo / Gossamer-style faint dust rings, a real-world precedent). Random breakup-gap roll skipped for WISPS so the dust haze stays pure. Combined with the new `ringOpacity` curve, post-tonemap visibility ladder is now: WISPS 0.05–0.23 (barely there) → DEBRIS 0.23–0.72 → URANUS 0.30–0.90 → SATURN 0.72–0.9999.
- **Keyboard dismiss centralised on `touchRipple`**: Added `dismissKeyboard: Boolean = true` parameter to `Modifier.touchRipple`. The modifier now captures `LocalFocusManager.current` and calls `focusManager.clearFocus()` before invoking `onClick()`. Default-on covers every existing ripple-driven button across the app (planet cards, filter button, regenerate, X-clear, back arrows, system search results) without per-callsite changes — the user-requested "tap any button to close the keyboard" behaviour for catalog/system pages naturally extends to anywhere ripples are used. The opt-out flag is reserved for buttons that should leave the keyboard up, though no current callsite uses it. `pushOnPress` + `clickable` patterns (filter pills, sort selector, text-label links) NOT auto-cleared — those are typically pressed when no text input is focused, and the explicit `clickable { }` lambda lets each callsite handle focus if needed. Avoids a global "tap anywhere outside text" interceptor that would have fought BasicTextField's own focus management.
- **`cachedPlanetEntry` only updates on a successful match (blank-planet-detail bug fix)**: Cross-system navigation (e.g. catalog → system A's planet, then star map → system B's planet) hit a window where `state.systemDetail` still held A's data while `state.selectedPlanetName` had already advanced to a planet name that only existed in B. The `LaunchedEffect` ran `det.planets.find { it.name == pn }` against A's list, got `null`, and unconditionally wrote that `null` into `cachedPlanetEntry`. The PLANET-page `if (entry != null)` gate downstream then gave up — blank screen until B's data finally loaded, by which point a re-emission would have re-resolved the entry. Fix: `det.planets.find { ... }?.let { cachedPlanetEntry = it }` — only update on a successful match. During the transition window we keep the previous entry (worst case a frame of stale planet data, never a blank page); when B arrives the LaunchedEffect re-fires with the new det and resolves correctly. Architectural alternative would have been to move planet-entry resolution into the ViewModel and key it on hostname-change, but the local-cache surgical fix matches the existing pattern for `cachedDetail` and `cachedOrbitalState`.

## Phase 10 — Candidates (2026-05-04)

- **NASA-tracked exoplanet candidates and confirmed false positives**: Alongside the existing confirmed-planet catalog. Off by default — opt-in via DATA setting. Phase 1 of the candidates plan (TOI ingestion).
- **Scope cap: TOI only for Phase 1**: New `toi` TAP table joins `pscomppars` (existing) and `stellarhosts` (existing). KOI deferred to Phase 2 (Kepler false-positive archive). Bookmarks + disposition-history tracking deferred to a separate plan. exoplanet.eu intentionally out of scope — single TAP endpoint, single ID convention, single sync pipeline beats marginal coverage of disputed claims.
- **Disposition collapsed to 3 user-facing values**: `Disposition.{CONFIRMED, CANDIDATE, FALSE_POSITIVE}`. TFOPWG's `APC` (Ambiguous Planet Candidate) folds into `CANDIDATE` — too rare (~5–10% of TOIs) to merit its own tab/badge, and reads to a user as "candidate, lower confidence" not as a meaningfully different bucket. Raw disposition strings preserved on `CandidatePlanetEntity.disposition` for completeness/debugging. `Disposition.fromRaw(String?)` normalises both TOI's `tfopwg_disp` (CP/KP/PC/APC/FP/FA) and KOI's `koi_disposition` (CONFIRMED/CANDIDATE/FALSE POSITIVE).
- **Sync policy: candidates always sync; toggle is display-only**: TOI rows ingest unconditionally during the existing `PlanetRepositoryImpl.sync()` — the user-facing `includeCandidates: Boolean` setting controls visibility (catalog sub-tabs + system page sections + planet badge), not data ingestion. Reasoning: TOI is ~7,500 rows × ~14 columns ≈ 1 MB on a fresh sync, marginal against the 6,150-planet baseline. Gating sync on the toggle would force a "toggle on → empty UI → trigger resync → wait" UX every time a curious user enables it. The toggle's question is "do I want to see this?", not "do I want to spend bandwidth?". Settings copy chosen as "Include unconfirmed candidates" rather than "Show ..." to signal it affects what's available system-wide, even though backing data is always fetched.
- **Defensive sync — TOI failure doesn't block planets**: `apiService.fetchToi()` is wrapped in a try/catch that logs and yields `null` on failure, mirroring the existing stellar-hosts fallback pattern. The transaction conditionally upserts candidates only when the fetch succeeded; on failure the local candidates table simply stays at its previous state. Avoids a single TAP-server hiccup taking down the user's confirmed-planet refresh.
- **TOI replace-all on every sync**: TOI dispositions can flip (PC → FP, FP → CP, etc.) and the table is small enough that `candidateDao.deleteBySource("TOI") + upsertAll(...)` inside one transaction is cheaper than diffing. Avoids stale rows when a candidate is removed from the upstream list. KOI (Phase 2) will likely use the same wholesale-replace pattern.
- **Separate `candidate_planets` table — no FK to stars**: Different ID space (TOI numbers like `1234.01`, not `pl_name`), different schema (transit-only — no atmosphere/orbit-derived data), different sync schedule, and many candidate hosts (TIC-only) don't exist in the `ps`-derived stars table. Foreign-key constraint would require pre-creating synthetic StarEntity rows or accepting orphans — both worse than just keeping candidates in their own table and reconciling at observe-time. Indices on `ticId`, `disposition`, `source`. DB version bump 10 → 11; existing `fallbackToDestructiveMigration(dropAllTables = true)` rebuilds on next launch.
- **TIC reconciliation done at observe-time, not at ingest**: `ObserveCandidatesUseCase` builds a `tic → Star` map from the live stars flow per emission, swaps the candidate's placeholder hostname (`"TIC <id>"`) for the matched primary's real hostname when found. Synthesises a stand-in `Star` from the candidate's per-row stellar fields (`hostTeffK`, `hostRadiusSolar`, `hostDistancePc`, `ra`, `dec`) for unmatched TICs. Keeping the swap in the use case rather than the entity means a future stars-table sync that newly contains a previously unmatched TIC retro-attaches existing candidate rows automatically — no migration, no resync.
- **TOI ↔ ps deduplication on (TIC, period within ±1%)**: A TOI row whose disposition is `CONFIRMED` and whose TIC + orbital period matches a row in the confirmed `planets` table is dropped from the candidate output — it would otherwise show as a duplicate of an entry already in the main catalog. Period match uses ms-precision rounding with a ±10ms band on the `(period × 1000).toLong()` integer key. Implemented as a Set lookup in `ObserveCandidatesUseCase.shouldDedup` for O(1) per-candidate cost.
- **`SystemPlanetEntry.disposition` carries the bucket through the pipeline**: Default value `Disposition.CONFIRMED` so existing call sites (ObserveCatalogUseCase, ObserveSystemDetailUseCase confirmed-planet path) compile unchanged. Candidate + FP entries pass the explicit value. Avoids a parallel `CandidateEntry` type with a duplicated SystemPlanetCard — UI re-uses the same card and reads `disposition` for the badge in PlanetDetailContent.

- **Sub-tabs in catalog over per-row badges**: 3 sub-tabs (CONFIRMED / CANDIDATES / FALSE POSITIVES), revealed only when `includeCandidates` is on. With the toggle off, the catalog is identical to the previous single-tab UI. Tab choice carries the disposition signal in catalog (no per-card badge — keeps the dense catalog row visually unchanged). Header noun reactively updates with the active tab: "5,012 exoplanets", "1,242 candidates", "3,891 false positives". Pill bar (`CatalogTabBar`) reuses the settings `OptionRow` styling — accentGoldSubtle bg + accentGold text on active, surfaceRaised + textSecondary on inactive — so it reads as part of the same visual system. `onTabChange` collapses any expanded card (the row key from the previous tab won't exist in the new one, so `expandedPlanetName` cleanup avoids a brief re-expansion when revisiting). Changing tab keeps search query + filters live across tabs — a deliberate choice; switching tabs is treated as a "look at the same query in another bucket" gesture, not a clean reset.

Avoids a stale state where the catalog renders zero rows because the now-hidden tab is "active". Toggling the master setting back on restores the user-selected tab from the still-stored MutableStateFlow.
- **Per-system sections in the System tab**: `PLANETS (n)` / `CANDIDATES (n)` / `FALSE POSITIVES (n)` headers, each rendered only when its bucket is non-empty (per plan: "show PLANETS (2) and FALSE POSITIVES (1) but not the CANDIDATES section" when there are no candidates). Cards re-use `SystemPlanetCard` unchanged across sections — the section header is the disposition signal in the system page, no per-card badge there either.

Pulled the planned Phase-2 work forward — the data path was already generic enough that adding a second source was small.   - TOI: `"TIC <id>"` placeholder, overridden by use case on TIC match
  - KOI with `kepler_name`: `stripKeplerPlanetLetter(kepler_name)` (e.g. "Kepler-186 b" → "Kepler-186")
  - KOI without `kepler_name`: `"KIC <id>"`

  Storing it in the entity rather than computing in the use case avoids running the regex strip + per-source branching on every emission. The use case still does the override-on-TIC-match step for TOI rows (those entities only get the placeholder at ingest).
- **Renamed `ticId` → `hostId` on the entity**: `ticId` was misleading once KOI joined — KOI rows store a KIC numeric in that column, not a TIC. Renamed to `hostId` (source-native catalog identifier; meaning is determined by the row's `source` value). Index also renamed. Indices added: `hostname` (new column, used by direct-hostname match for KOI in the use case) and the existing `hostId`, `disposition`, `source`. DAO query `observeByTicId(ticId)` renamed to `observeByHostId(hostId)`.

The case where two different confirmed planets share a hostname AND a period within 10ms is essentially nonexistent in practice (different planets have very different periods within the same system).

For a `Kepler-186` candidate, the matched primary is the existing `Kepler-186` star already in our `ps`-derived stars table, populating its full stellar parameters (Teff, radius, mass, metallicity, distance, etc.) on the rendered candidate card. For unmatched KOIs (rare — Kepler hosts are well-catalogued in `ps`), the synthetic Star uses the per-row `koi_steff` + `koi_srad`. RA/Dec carries through; distance is null (KOI has no distance column).
- **`displayName` always `<source>-<sourceId>` for stable PK and predictable copy**: Considered using `kepler_name` as the display name when present (e.g. "Kepler-186 b" instead of "KOI-K00752.01"), but rejected because (a) the row's `candidateId` PK has to stay stable across promotions, so we'd be storing one ID and displaying another, and (b) when a Kepler-named row leaks past dedup (rare; happens when a confirmed-planet entry exists under a different host hostname), displaying it as "Kepler-186 b" in the candidates tab would be confusing. "KOI-K00752.01" makes the row's status clear — it's a KOI archive entry, not a confirmed one. Users who want the friendly name go to the confirmed catalog.
- **Catalog tab bar redesigned as conjoined bar with sliding highlight**: Original Phase-1 implementation was three separate pill buttons in a Row with gaps. User feedback: should read as a single segmented control. New `CatalogTabBar` is a single rounded rectangle (radius 20dp) at `surfaceRaised` background with `divider` border, containing three equal-weight click targets and a sliding highlight that animates between segment positions over 260ms `FastOutSlowInEasing`. Highlight uses `accentGoldSubtle` bg + `accentGoldBorder` with a 2dp inset from the outer bar so the gold pill reads as a separate element rather than overlapping the outer border. Text colour also animates between `textSecondary` (inactive) and `accentGold` (active) on the same 260ms curve. Labels uppercased to match the section-header convention used elsewhere in the app (PHYSICAL PROPERTIES, ORBITAL PARAMETERS, etc.).
- **Tab-bar highlight painted via `Modifier.drawBehind` (not a sibling Box)**: First implementation used a sibling `Box` with `.fillMaxHeight()` inside `BoxWithConstraints` to draw the sliding highlight. The highlight didn't render at all — the parent `BoxWithConstraints`' height is determined by content (the labels Row), but `fillMaxHeight()` on the highlight resolves at measure time before the Row's intrinsic height is known, so the highlight measured to zero and painted nothing. Switched to `Modifier.drawBehind` on the bar itself: paint after `.background(surfaceRaised)` and before `.border(divider)` in the modifier chain, with a single `animateFloatAsState`-driven index that places `drawRoundRect` calls at `segmentWidth × index + 2dp` inset. Sidesteps the layout cycle entirely — the bar's `size.width` and `size.height` are correct at draw time. Children (label Row) draw on top of the highlight via the normal Box content slot.
- **KOI display name strips the leading "K" + zero-padding**: First KOI rollout used `"KOI-${kepoi_name}"` directly, which yielded "KOI-K00752.01" — the `kepoi_name` column already has a leading `K` (Kepler-mission catalog tag) and is zero-padded to 5 digits. NASA's published convention is `KOI-752.01`. Added `normaliseKoiId()` in `DtoToEntityMapper` that strips the leading `K` and any leading zeros from the integer portion before the dot, preserving the planet-index suffix as-is. Display name is now consistent with how NASA, ExoFOP, and astronomy literature reference KOIs. Substring search ("752.01") matches both the candidate row and a human typing the canonical short form.
- **KOI source switched from `q1_q17_dr25_koi` → `cumulative`**: DR25 is a frozen pipeline release (final Kepler mission output, June 2017). The `cumulative` table is the live current-state catalog NASA recommends for general use — it carries dispositions updated post-DR25, late-added KOIs, and any revisions from follow-up vetting. User reported a known KOI by number wasn't showing up in our list; switching to `cumulative` is the right default. Trade-off: `cumulative` rows can include slightly different parameter values than DR25 (the cumulative pipeline takes the most recent reduction), but for browsing-the-catalog purposes that's a feature not a bug.
- **Candidate classification decoupled from System tab navigation (~500ms freeze fix)**: User reported a ~500ms lag when entering the planet info screen from any source (catalog, star map orbital "View planet"), and a similar pause right after the zoom-in animation when entering the orbital screen from the star map. Symptom: the previous page lingers for the duration before the new page paints. Root cause: `ObserveSystemDetailUseCase` had a constructor dependency on `ObserveCandidatesUseCase`, whose flow ran `PlanetClassifier.classify` + `VisualProfileEngine.generate` over **all ~17,000 candidates** every time it emitted. Each navigation triggers a fresh `flatMapLatest` invocation, which fresh-subscribes to the candidate flow, which re-runs the full-list classification before the system page can render its first frame. Fixed by:
  1. Extracting the per-candidate work into a stateless `CandidateClassifier` helper (`buildIndex`, `resolveStar`, `shouldDedup`, `classify`)
  2. `ObserveCandidatesUseCase` keeps its full-list classify (catalog needs every candidate up-front for the CANDIDATES + FALSE POSITIVES sub-tabs)
  3. `ObserveSystemDetailUseCase` no longer depends on `ObserveCandidatesUseCase` — it observes raw `repository.observeAllCandidates()` directly, runs `filterCandidatesForHost(hostname)` to get the small subset (typically 0–10 per system), then classifies only those rows inline
- **Single classifier path for catalog and system detail**: Both consumers now share the same `CandidateClassifier` helper, so the per-candidate visual + classification output is bit-for-bit identical between the catalog cards and the system page candidate cards. No risk of divergence as the classification logic evolves — change the helper, both consumers update. The `Index` data class precomputes `(starsByTic, starsByHostname, confirmedByHostnamePeriod)` once per emission and threads it into per-row calls, so the O(n × stars) and O(n × planets) scans aren't repeated for each candidate.
- **Tab-bar fill background dropped — highlight reads washed out on `surfaceRaised`**: First conjoined-bar rollout used `.background(colors.surfaceRaised)` to paint a filled mid-tone rectangle, with the gold-subtle highlight painted on top. User reported the highlight (both fill and border) looked "too light, as if applied twice." Cause: `surfaceRaised` (0xFF151929, RGB ~21,25,41) is ~3× more luminous than the catalog page bg (0xFF06080F, RGB ~6,8,15). A 7%-alpha gold composited on the lighter base lifts the perceptual luminance of the highlight area from ~19 to ~36, turning a tinted "selected" feel into a washed-out pale rectangle — and the dark-purple `accentGoldBorder` (0xFF2A1F35) on top of that lighter base lost contrast too. Same accent tokens read with full saturation in settings' `OptionRow` because those pills sit directly on the dark page bg. Fix: remove the bar's filled bg entirely. The bar is now a thin `divider`-bordered rectangle with the gold-subtle highlight painted via `drawBehind` directly on the page bg. Outer border ties the three segments together visually as a "conjoined bar" without needing a fill. Saturation now matches OptionRow.

## Phase 11.1 — Bookmarks foundation (2026-05-04)

Phase 1 of [PLAN_BOOKMARKS.md](PLAN_BOOKMARKS.md) — foundation for tracking parameter changes between syncs. End state: user can bookmark/unbookmark from the catalog card expanded panel and filter the catalog to bookmarked-only via the new header button + "Saved only" pill. Diff tracking, info-page bookmark toggle, and update badges arrive in Phase 11.2.

- **Separate `bookmarks.db` Room database**: User data lives in its own database, distinct from `tadmor.db`. The catalog DB uses `fallbackToDestructiveMigration(dropAllTables = true)` because catalog data is replaceable from NASA's archive — but bookmarks aren't. Putting bookmarks in their own DB lets the catalog DB stay free to wipe across schema changes while bookmarks get proper migrations from v1 onwards. Trade-off: two DBs to manage. Worth it — the alternative is committing to writing migrations for every catalog table change going forward, which slows iteration on the catalog side.
- **Single-table schema, JSON snapshot column**: `BookmarkEntity(planetKey, source, bookmarkedAt, snapshotJson)`. Snapshot stored as a kotlinx-serialised `PlanetSnapshot` JSON string rather than fanning out to columns. Reasoning: snapshots aren't queried, only deserialised whole at diff time; schema stays trivially evolvable (new fields auto-roundtrip with `ignoreUnknownKeys = true`); ~50 rows × ~300 bytes of JSON each is negligible storage. The flat-fields alternative would require a column migration on every new diffable field, which loses the schema-evolvability benefit.
- **`PlanetSnapshot` — flat record of diffable fields, not the full Planet/CandidatePlanet**: A focused record of the dozen-ish parameters surfaced by the planet info page property grids, plus a `disposition: String` so candidate→confirmed transitions register as updates (the headline reason a user would bookmark a candidate). Star fields intentionally omitted — Phase 11 only diffs planet-level parameters; star tracking is out of scope until we revisit "system bookmarks" as a feature in its own right.
- **`Planet.toSnapshot()` and `CandidatePlanet.toSnapshot()` extensions**: Both sources project to the same `PlanetSnapshot` shape. `Planet.toSnapshot()` defaults disposition to `CONFIRMED.name`; the `CandidatePlanet` variant uses the candidate's actual disposition. Confirmed planets that always have null mass/SMA/eccentricity round-trip as null on the snapshot, matching the candidate path — they never register as a "change" until the candidate gets promoted to a row in the confirmed `planets` table with measured values.
- **`kotlinx.serialization` plugin added to domain module**: Required so `PlanetSnapshot` can carry `@Serializable` directly (rather than a translator-pattern DTO in data). Pure-Kotlin lib, no Android dependency, fits the domain module's "pure Kotlin, no Android dependencies" rule. Same plugin already lived in `data` for the TAP DTO; reusing the existing version constraint keeps Gradle clean.
- **Unified `planetKey` across confirmed + candidate sources**: PRIMARY KEY is just the planet display name (`Planet.name` for confirmed, `CandidatePlanet.displayName` like "TOI-1234.01" / "KOI-752.01" for candidates). NASA's namespaces don't collide — confirmed planets always carry a planet letter ("Kepler-186 b"), candidate IDs are always source-prefixed numerics. Keeping a single primary key column avoids a (source, id) compound key and the unique-constraint juggling that goes with it.
- **`source` column for live-data lookup, not for primary key**: When the diff phase needs to fetch the current snapshot for a bookmarked planet, it consults `source` to know which table to query (planets vs candidate_planets). Without the column we'd need to scan both for every diff. Cheap denormalisation that pays off once per emission.
- **`ExoFilterButton` collapsed to icon-only square**: The user's new bookmark-button beside it needs space, and the redundant "Filters" label was the easiest sacrifice — the button's active-state gold tint plus the active-filter chips below already signal "filters are on". Square footprint matches the bookmark button's footprint so the two read as a pair. Internal icon size bumped from 14dp/18dp to 18dp/22dp to compensate for losing the text — keeps the button visually balanced when there's no label beside the icon.
- **`BookmarkFilterButton` mirrors `ExoFilterButton`'s shape and active-state colours**: Same square `RoundedCornerShape(20.dp)` outline, same `accentGoldSubtle` bg / `accentGoldBorder` outline / `accentGold` icon when active, same `divider` outline / `surfaceRaised` bg / `textSecondary` icon when inactive. The hollow→filled `BookmarkIcon` swap is the only difference visually, so the pair reads as "two related toggles" rather than "two unrelated buttons".
- **`BookmarkIcon` Canvas-drawn shape, not a vector drawable**: Per the standing rule (no icon libraries / pre-baked vector assets — all icons are stroke-based custom). Single ribbon-with-V-notch path, ~0.66 wide × 0.92 tall inside its bounds for visual margin. `filled = false` strokes; `filled = true` fills. 1.2dp stroke matches the 20dp-icon convention used by other custom icons in the app (DESIGN.md §5).
- **`SavedOnlyChip` is a fixed-content pill, not parameterised by `ActiveFilterChip`**: Its label is hardcoded "Saved only", icon is hardcoded `BookmarkIcon`, and it always renders even when there are no other active filters. Trying to fold it into `ActiveFilterChip` would require icon-slot + custom-text plumbing for one usage site. Cleaner to ship as a sibling component that shares the active-filter pill's visual rules but has its own narrow API.
- **"Saved only" always at index 0 of the active-filter row**: The `LazyRow` items list explicitly prepends the SavedOnly chip when active, before iterating the regular `activeLabels`. Keyed via `"savedOnly"` so the existing animateItem fade-in/out plays when the user toggles the bookmark filter, matching how regular filter chips behave.
- **"Clear all" also turns off the saved-only filter**: Logical extension — saved-only is conceptually a filter, so "clear all filters" should drop it too. Implementation: `ClearAllChip.onClick` calls both `viewModel.onFilterChange(state.filterState.clear())` AND `viewModel.onSavedOnlyToggle()` when the latter is active. Matches the user mental model of one button that resets everything.
- **Card bookmark toggle uses push-only feedback (no ripple)**: The expanded-panel bottom row is `[Save] ........ [View in System >]`. "View in System" was already push-only (no ripple). Making the bookmark side ripple too would have made the symmetric pair behave inconsistently. Push-only on both keeps the visual rhythm tight.
- **Card bookmark toggle label flips text content, not just icon state**: Hollow icon + "Save" when inactive; filled icon + "Saved" when active. Two redundant signals (icon + text) for a small interaction is cheap clarity — the user shouldn't have to remember whether hollow means "tap to save" or "currently saved" by reading the icon shape alone.
- **`isBookmarked` set passed top-down through `CatalogUiState`, not looked up per row**: ViewModel exposes `bookmarkedKeys: Set<String>` once per emission; cards do an O(1) `name in keys` check. Avoids per-card collectAsState subscriptions to a per-key flow, which would explode subscription count for large catalog renders. Set membership is O(1) with no per-card overhead.
- **CatalogViewModel decides `source` from the entry's disposition + name prefix**: TOI vs KOI is read from the candidate display name's prefix (`KOI-` vs anything else, since TOIs are `TOI-1234.01`). Could in theory check the candidate flow's source field via a join, but the prefix is already canonical and stable in the displayName by construction. One source-of-truth lookup at bookmark-add time.

## Phase 11.2 — Diff tracking (2026-05-04)

End state: full feature loop. Bookmark a planet, sync, see "n updated" on the card and the badge on the filter button, open the planet → diff display, close → snapshot resets.

- **Diff computed on demand, not stored**: `computeDiff(snapshot, current): PlanetDiff` runs on every catalog/system flow emission. No `is_dirty` flag column on the bookmarks table — the snapshot itself carries the comparison baseline, and "no diff" is simply `snapshot == current`. Keeps the bookmarks DB write-light: only `add`, `remove`, and `consumeUpdates` mutate it.
- **`PlanetDiff.changes` is a list of [FieldChange], not a flat record of typed nullable old/new pairs**: Two reasons. First, it makes "iterate the changes for the badge" the natural code path — `diff.changes.size` is the badge count, no need for a custom counter. Second, it lets the UI render a subset of changes (e.g. property-grid rows show only their tracked fields; the disposition banner picks the DISPOSITION change off the same list). A struct with all fields would force every consumer to walk all 13 fields.
- **`PlanetField` enum at the domain layer carries `displayLabel`**: The label is the abstract "what users call this parameter" string (e.g. "Longitude of periapsis"). Presentation layer has its own column-budget-aware mapping (`propertyLabelFor`) that produces UI-specific shorthand ("Long. of periapsis"). Keeping these separate means the domain model isn't shaped by current UI string choices, and a future settings-toggle that flips between full and abbreviated labels wouldn't need a domain change.
- **Diff projected as `Map<String, String?>` keyed on UI label, not by [PlanetField] directly**: Property grid is a generic component — passing `Map<String, String?>` (label → formatted old value) keeps it ignorant of `PlanetField`, which is a domain type. Trade-off: one extra label-mapping switch (`propertyLabelFor`) at the call site. Worth it: the grid stays reusable and doesn't accrue domain dependencies.
- **Disposition diff surfaces as a banner above the property grids, not as a row inside one**: A disposition flip (TOI candidate → confirmed planet, confirmed → false positive) is the headline reason a user would bookmark a candidate in the first place. Burying it in a property-grid row with the same visual weight as a 2% radius update would lose the signal. The banner gets `accentGoldSubtle` background + `accentGold` text, sits above the PHYSICAL PROPERTIES section, reads "Was: Candidate → Now: Confirmed".
- **Diff display uses two-line stack, not single-line "old → new"**: First sketch was one-line `<old>  →  <new>`. Property grid columns are half-width — long values like "0.3624 → 0.3814" overflow. Switched to a vertical stack: struck-through old on top, gold "→ new" below. Reads top-to-bottom as "was X, now Y", and each line gets the column's full width budget.
- **`accentNotification` (#D45050) distinct from `accentFalsePositive` (#C25A4A)**: User feedback during planning. Notification red wants "new attention" semantics (fresh updates worth checking); FP red wants "negative outcome" semantics (this candidate didn't pan out). Same hue family but the notification one is more saturated so it pops as a badge against the gold-tinted button bg.
- **Catalog card "n updated" replaces the data-availability indicator slot**: When `updateCount > 0`, the right-side slot in the subtitle row renders "$n updated" in `accentGold` instead of the dot indicator. When `showDataIndicator` is off in settings, the slot stays empty until updates exist — the slot is reserved for the indicator's footprint regardless. After consume-on-close clears the diff, `updateCount` drops to 0, and the indicator (if enabled) reappears automatically. No animation between the two — the diff/indicator presence is data-driven, and AnimatedContent in a list-item slot would have animated on every emission (noisy).
- **Card gold left edge painted via `drawBehind`, after surface fill**: 2dp wide accent bar at the card's left edge, full card height, `accentGold` colour. Drawn after the surface fill in the modifier chain so it lands on top of the rounded-corner clipping. A glance signal across a scrolling catalog — easier to spot than the (small) gold "n updated" text alone, especially with multiple updated bookmarks visible at once.
- **`BookmarkFilterButton` badge: rounded-rect, top-right, ~30% beyond the corner**: Two-digit counts read cleanly with rounded-rect (16dp min height) instead of perfect circle. Offset (4dp, -4dp) pulls the badge ~30% beyond the button's corner so it doesn't crowd the bookmark icon inside. White text on `accentNotification` red — high contrast at small sizes.
- **`SystemViewModel.consumeOldPlanetIfChanged()` invoked from every site that mutates `selectedPlanetName`**: Hardware back, GUI back arrow, planet → planet switch, `resetToSearch()` while in PLANET mode. NOT invoked on tab switch (`leavingTab` flow doesn't go through these), NOT on process death. The function is idempotent and self-checks: if `old == newPlanetName` (no-op transition) or `old !in bookmarkedKeys` (not a bookmark) or no matching SystemPlanetEntry in the live data, it returns without writing.
- **`PlanetBookmarkToggle` AnimatedContent keyed on `(planetKey, isBookmarked)` with `contentKey = { it.first }`**: The user wanted cross-planet navigation to fade the bookmark state in/out at 220ms, but bookmark toggles within the same planet should update the icon immediately (no fade). `targetState = (planetKey, isBookmarked)` so AnimatedContent's content lambda receives the up-to-date isBookmarked. `contentKey = { it.first }` means AnimatedContent only treats planet-key changes as a transition; same-planet bookmark toggles re-invoke the content lambda but skip the fade.
- **`CatalogEntry.toSnapshot()` extension wraps `Planet.toSnapshot().copy(disposition = ...)`**: Avoids the disposition mismatch where a candidate row's synthesised Planet has its disposition default to CONFIRMED. One call site (catalog bookmark add) used to do the .copy inline; now both catalog and system-tab bookmark adds funnel through this helper. Single source of truth for "what's in a bookmark snapshot".

## Phase 11.3 — Bookmarks debug + polish (2026-05-04)

Closes Phase 11. Adds the developer-facing simulation toggle, animation polish on the bookmark UI, and documents the edge-case behaviours that fell out of the existing flow logic for free.

- **`simulateBookmarkUpdates: Boolean = false` in `UserSettings`**: Persistent toggle (lives in DataStore alongside the other display preferences) rather than a one-shot button. Persistent matches how the user described the workflow: "the next sync" — every sync while the toggle is on injects a fresh round of fake changes, so iterating on diff-display visuals is just "toggle on, refresh, look, refresh again, look at the new diff". A one-shot button would require re-tapping between every iteration.
- **DEBUG section at the bottom of settings**: New `SectionHeader("DEBUG")` after DATA. Section ordering communicates intent — debug utilities go last so accidental discovery is unlikely; users browsing settings hit the regular preferences first. Toggle copy explicitly notes "Only the snapshot is touched — real catalog data is unaffected" so a user who flips it on doesn't worry about corrupting their archive.
- **Simulation mutates the bookmarks DB, never the catalog DB**: `BookmarkRepository.simulateUpdates()` reads each bookmark, decodes its snapshot JSON, picks 1–2 fields, scales numeric values by a random factor in [0.85, 1.15] OR flips disposition (~30% chance), and writes the mutated snapshot back. The catalog's planets / candidates / stars tables are untouched — when the user resyncs, the fresh upstream data is intact. This makes the simulation reversible (re-bookmarking captures the current snapshot, clearing the fake "old" values). Trade-off: the diffs are narrower than real catalog updates would produce — no "data added" scenarios since we skip null fields. Acceptable for debug; demoing all field types isn't the point.
- **Disposition flip restricted to ~30% probability per bookmark per sync**: Every simulation run pulls 1–2 mutators from a pool that always includes a disposition-flip option. If we ran the disposition flip every time, every bookmarked candidate would flip disposition every sync — the disposition banner would dominate every view. The 30% gate keeps disposition changes feeling like "occasional events" alongside the more common numeric tweaks.
- **Sync hook in BOTH `CatalogViewModel.onRefresh()` and `DataSyncWorker.doWork()`**: Manual pull-to-refresh and the periodic background sync both honour the toggle. Behaviour is identical between the two so a user testing with manual refresh sees the same update injection pattern that production users would experience if they had the same toggle on (which they wouldn't — it's debug-only). Consistency reduces "works in dev, broken in prod"-style surprises if the simulation logic ever moves to a "real" code path.
- **Hidden bookmark behaviour falls out of the existing diff-projection logic for free**: Plan listed "hide bookmarks whose `planetKey` no longer exists in the live catalog" as a Phase 11.3 task. Inspection showed it was already correct: `CatalogViewModel.diffsByKey` `mapNotNull`s with `entriesByKey[bookmark.planetKey]?.toSnapshot() ?: return@mapNotNull null` — bookmarks for missing entries are silently skipped from the diff map, so they don't contribute to `unreadUpdatesCount` or the gold left edge. The "Saved only" filter applies on the source list (`it.planet.name in keys`), so a bookmark for a missing planet doesn't fabricate a row. The bookmark stays in the bookmarks DB (we don't auto-delete user data) but is invisible UI-side. No new code needed.
- **`BookmarkFilterButton` accent transitions animate at 220ms `FastOutSlowInEasing`**: Background, border, and icon colours all `animateColorAsState` between the inactive (`surfaceRaised` / `divider` / `textSecondary`) and active (`accentGoldSubtle` / `accentGoldBorder` / `accentGold`) states. Same curve as the catalog tab bar text-colour fade and settings option-row press animations, so the bookmark toggle reads as part of the same visual system rather than a one-off control with its own motion language.
- **Hollow→filled icon `Crossfade` over the colour fade, same 220ms**: BookmarkIcon's `filled = Boolean` flips between two completely different stroked-vs-filled paths. A snap looked jarring after the colour transitions started animating. Wrapping the icon in `Crossfade(targetState = isActive, ...)` with the same 220ms fades both variants through 50%-alpha at midway; combined with the simultaneous colour animation, the visual effect reads as the icon "filling in" rather than "swapping out".
- **Badge entrance/exit uses `AnimatedVisibility` with fade + scale**: When `unreadCount` transitions 0 ↔ N, the red notification badge fades in/out (220ms enter, 180ms exit) AND scales between 0.6 and 1.0. The slight scale gives the appearance an "arrival" feel — fits the semantic of "new attention needed" — and the asymmetric durations (faster exit) mean dismissal feels lighter than discovery. Pure fade alone made the badge feel ghostly; pure scale alone made it feel mechanical.
- **Card-level Save/Saved toggle uses single `Crossfade` wrapping the icon + label row**: One Crossfade keyed on `isBookmarked` covers both the BookmarkIcon's `filled` state and the BasicText's "Save"/"Saved" content. Scoping a single crossfade over both children keeps the two changes visually unified (single 220ms fade), avoiding the staircase effect of separate per-element transitions firing at slightly different times. Push animation on tap remains the primary feedback signal; the crossfade is supporting visual polish.
- **Bookmarks DB schema export already enabled — no Phase 11.3 work needed**: `@Database(... exportSchema = true)` was set on `BookmarksDatabase` at v1 (Phase 11.1). KSP arg `room.schemaLocation` already points at `data/schemas/`. The first build emits `BookmarksDatabase/1.json` automatically; from v2 onward, proper migrations are mandatory (no `fallbackToDestructiveMigration` was ever added to the bookmarks DB builder). User data protection contract holds.

## Phase 10/11 polish (2026-05-04)

- **Candidate dedup no longer requires CONFIRMED disposition**: User reported TOI-2104.02 (TFOPWG disposition: planet candidate) showing up alongside TOI-2104 b (confirmed in `ps`) at the same 5.9-day period — clearly the same body, but `shouldDedup` was gating on `candidate.disposition == CONFIRMED` and skipping it. TFOPWG dispositions don't reliably update when a body becomes confirmed elsewhere — the `.02` suffix often persists with PC/APC long after the planet is in `ps`. Dropped the disposition check; now ANY candidate whose (hostname, period within tolerance) matches a confirmed planet is hidden from the candidates view. The (host, period) signature is unique enough across NASA's catalogues that false positives are essentially impossible — two distinct planets at the same star with periods within 1% of each other don't exist in the data.
- **Period-match tolerance changed from fixed ±10 ms to ±1% relative**: The fixed band was pathologically tight at long periods — ±10 ms is ±0.003% of a 365-day orbit, and ±0.17% even at 5.9 days, both narrower than the typical reduction-to-reduction drift between TOI/KOI and `ps` for the same body. Switched to `abs(candidatePeriod - confirmedPeriod) < period × 0.01`. Index reshaped from `Set<Pair<String, Long>>` (hostname + period-in-ms key, O(1) lookup) to `Map<String, List<Double>>` (per-host list of confirmed periods, O(p) scan where p ≈ 1–10 per system). The linear-scan cost is negligible because per-host planet counts are tiny; the relative-tolerance correctness benefit is large.
- **HW back consume race fixed by capturing state synchronously before launching the write coroutine**: Hardware-back paths from PLANET (esp. `resetToSearch()` from a fromCatalog navigation) null `selectedHostname` synchronously, which makes the `systemDetail` Flow re-emit null. The original `consumeOldPlanetIfChanged` launched a coroutine that called `systemDetail.first()` AFTER those mutations had happened — the flow returned null, the coroutine bailed, no consume. Fix: read `uiState.value` (StateFlow, sync) for the entry data BEFORE any state mutation, then launch a coroutine that only does the DB write. The launched coroutine no longer reads any reactive state. GUI back already worked because `onGuiBackFromPlanet` doesn't touch `selectedHostname`, so the flow stayed valid — same code path, but resilient to upstream resets now.
- **Catalog "n updated" badge: missed `SubtitleRow` call site**: `PlanetCard` calls `SubtitleRow` twice (accessible-mode path and standard-mode path). The Phase-11.2 wiring added `updateCount = updateCount` to the accessible call but left the standard call using the default `updateCount = 0`. Standard-mode users (most users) never saw the badge because of that. Fixed by passing the param at both call sites — defensible to keep two layouts but they need to stay in sync.
- **System-tab planet cards: gold left edge + "n updated" added**: User wanted parity with the catalog card so a bookmarked planet flagged as updated reads the same in both places. New `updateCount: Int = 0` parameter on `SystemPlanetCard`, gold edge via `drawBehind` matching the catalog card's pattern. New `updateCountsByKey: Map<String, Int>` projection on `SystemUiState` (computed in `SystemViewModel.computeUpdateCountsByKey` once per emission across this system's planets/candidates/false-positives), threaded through `SystemDetailContent` to all three SystemPlanetCard call sites (planets, candidates, FPs).
- **System-tab "n updated" sits BELOW the class badge, not beside it**: First placement put the gold "n updated" text in the same row as `[name][badge]`, which forced the layout to give up width budget on every card render and crowded long planet names. Moved to a dedicated row directly below the class badge, right-aligned in standard mode (lands directly under the badge), left-aligned in accessible mode (matches the badge's left alignment). Mirrors the catalog card's vertical ordering where the indicator sits one row below the name+badge.
- **Expanded planet-card animation matched to LazyColumn item-placement timing**: Cards expanded with `tween(200)` linear, but LazyColumn's `animateItem(placementSpec = tween(260, FastOutSlowInEasing))` shifts sibling cards below it for 260 ms. The expanding card's content "outran" the items moving down — visible as the expanded panel content sliding past the panel's growing border, then snapping back when the placement animation finished. Fixed by aligning both to `tween(260, FastOutSlowInEasing)`. The discrepancy was invisible when the expanded card was the only result or at the bottom of the list (nothing to animate below it).

- **Star corona / glow + optical flare moved to in-front-of-planet rendering**: User wanted parity with the diffraction spikes — the corona should also occlude when the sun goes behind the planet limb or off-screen, and shrink + dim under the same `sunOcc` thresholds as the spikes. Pulled the `glow * starLight * 0.15 * haloAtten` and `opticalFlare * uSunColor * uSunIntensity * 0.5 * haloAtten` terms out of both the airless-planet sky path and the atmosphere sky path; centralised them in a new `computeStarCorona(rayDir, angle, sunDot, haloAtten)` helper. The helper is invoked from `writeFinal` directly, AFTER the planet rendering, AFTER ring/atmosphere mixing, BEFORE exposure scaling — so it lands on top of everything except the spikes (which paint atop the corona). The sun DISC stays on the sky path (unchanged) because it's geometry, not glow — physically correct for it to be occluded by the planet when behind the limb.
- **Corona shrinks AND dims with `sunOcc`**: The `glowSpread` in `exp(-distanceToEdge * spread)` is divided by `sunOcc` (clamped to 0.05 lower bound) — sub-1.0 sunOcc widens the spread, narrowing the perceptual extent of the glow. Final brightness multiplied by `pow(sunOcc, 2.5)` matching the spike brightness curve. Same `computeSunOcclusion() * computeScreenEdgeFade()` thresholds as the spikes, so corona / spikes / edge-fade transition together as one coherent visual.
- **Diffraction spikes blurred further (third pass)**: User reported the cores still read as defined rays at typical viewing distances. Lowered the `pow(|sin(...)|, exponent)` exponents another step across all 9 noise terms — cores from 3–8 to 2.0–5.0, halos from 1.0–1.8 to 0.7–1.2. Lower exponents widen each ray and soften the falloff toward the inter-ray gaps; the cumulative max stays around the same so brightness is preserved while the overall texture reads more diffuse-haze, less laser-line.
- **Corona disc-interior mask added (`smoothstep(drawRadius × 0.9 → × 1.1)`)**: After moving corona to writeFinal, user reported a solid bright disc visible through the occluding planet at close range. Cause: `glow = exp(-max(0, angle - drawRadius) * spread)` evaluates to `exp(0) = 1.0` for any pixel inside the disc area (where `angle <= drawRadius`), creating a bright plateau across the disc interior. `opticalFlare = exp(-angle * spread)` has the same behaviour — peak at `angle = 0`. On the old sky path this was invisible because the disc itself paints on the same pixels, hiding the corona behind it; in writeFinal the corona is in front of the planet, so the plateau leaks through the silhouette. Fixed with a `discMask = smoothstep(drawRadius * 0.9, drawRadius * 1.1, angle)` multiplying both glow and opticalFlare — zero inside the disc, full outside, with a soft 20 %-of-radius transition at the edge. Visually: corona becomes a halo around the disc's edge instead of a filled disc-with-halo. The actual disc is still rendered (correctly behind the planet) on the sky path, so end-to-end the user sees disc + halo when not occluded, just halo around the silhouette during partial occlusion, and nothing when fully occluded.
- **Corona disc-mask reverted; angle-based exponential gives a true point-light peak**: User reported the disc-mask version read as a "negative mask" — a hollow ring around the planet's silhouette during partial occlusion, with no bright centre. Real partial occlusions of bright point sources don't look like that; they look like a soft glow with a peak still visible behind the silhouette, fading outward. The actual fix wasn't the mask but the curve shape: the old `exp(-max(0, angle - drawRadius) * spread)` had a flat plateau across the disc area, AND that plateau leaked through occluding planets. Switching to plain `exp(-angle * spread)` removes the plateau entirely — brightness peaks at `angle = 0` (the sun direction) and decays smoothly outward through the halo. With no plateau there's nothing flat to leak through, and during partial occlusion the corona reads as a soft Gaussian-like glow with bright centre fading to halo edge — recognisable as a point light. The `sunOcc`-based shrink-and-dim still applies (size via `spread × occInv`, brightness via `pow(sunOcc, 2.5)`), so fully-occluded suns still produce zero corona. The disc itself stays on the sky path (correctly occluded by the planet); the corona's bright centre layers on top.

## Phase 12 — Globe renderer optimisation (2026-05-05)

Plan recorded in [PLAN_OPTIMISATION.md](PLAN_OPTIMISATION.md). User flagged two perf issues: a ~50–150 ms entry-frame stall when navigating to a planet page (across both airless and atmospheric bodies), and degraded framerate during atmosphere rendering — especially close-zoom on Earth-likes and full-screen hot sub-Neptunes. Root causes identified as (1) two synchronous bake passes running back-to-back in the first frame after navigation, totalling ~9 megafragments of high-octave noise, and (2) the 16×4 atmosphere ray march running unconditionally on every atmospheric pixel with early termination disabled. Fix lands across five steps; 1+2 first.

- **Atmosphere ray-march early termination re-enabled, but ground-hit-only at a tighter 0.5% threshold**: The original break condition `viewTransmittance < 0.03` was disabled "for testing" — left commented out at line 1167. User recalled an early-Phase-7 issue with the terminator turning black, almost certainly caused by that aggressive cutoff: limb / sky-only rays carry the brightest forward-scattered Mie contribution (sunset glow), and they're the longest-path rays that would trip a 3% threshold first. Re-enabled with two safety changes: (a) gated on `hitGround` so only ground-bound rays are eligible — sky/limb rays continue the full integration to preserve the limb glow, and (b) threshold dropped to 0.005 (well below the Reinhard tonemap's response floor) so even ground-hit rays only break once the remaining contribution is invisible. Expected savings: 25–40% inner-loop iterations on dense-atmosphere ground pixels (sub-Neptunes), with no visible change to limb / terminator rendering.
- **Cloud bake resolution dropped from 4096×2048 to 1024×512 (16× fewer fragments)**: The cloud noise texture was sized for crispness but cloud features are inherently softer than surface terrain — the same 1024×512 that satisfies surface-bake bumps comfortably exceeds cloud needs. Three sanity checks before the change: (a) LINEAR mag filter handles bilinear smoothing on the lower-res texture, (b) the cloud-normal gradient eps (0.01 in planet.frag, computing the bumped cloud surface) sits at ~1.7× the equirectangular pixel size (2π/1024 ≈ 0.006 rad at the equator), so gradient sampling stays correct, and (c) inspection of the bake shader confirmed it doesn't rely on per-texel sharpness — 5-octave warped FBM produces continuous output dominated by the lowest octave's scale. Cloud bake fragment count drops from 8.4M to 0.5M; combined with the surface bake (already at 1024×512), total entry-frame fragment work drops from ~9M to ~1M — eliminating most of the visible spike on its own.
- **Early-termination `break` was being compiled out (compiler unrolling)**: Bumped the threshold from the planned 0.005 to 0.03, then to 0.3, with no observable visual or performance change. The `#define iSteps 16` was letting the GLSL compiler fully unroll the outer loop and lower the `break` into predicated writes — the GPU was running all 16 step bodies regardless and discarding late results. Diagnostic confirmed by hard-coding `iSteps` to 8: framerate jumped substantially on hot sub-Neptunes and close-zoomed Earth-likes, with visible banding on thin atmospheres at 8 steps. Conclusion: outer view-ray loop IS the dominant fragment-shader cost, but capturing the savings requires the loop bound to be non-constant so the compiler can't unroll. Threshold left at 0.03 for when step 4 makes that switch.
- **`iSteps` / `jSteps` converted from `#define` to uniforms `uISteps` / `uJSteps`, driven from PlanetRenderer per-frame**: Atmosphere-thickness-driven step count: `>300 km → 16` (sub-Neptune envelopes keep full quality), `>80 km → 14` (Earth-like / Venus), `>0 km → 14` (thin / Mars-like — initially 12 but banding showed on the limb at that count). Inner loop scales `>80 km → 4` else `3`. Two compounding savings: (a) thin / mid atmospheres directly run fewer iterations, and (b) the early-termination `break` at the end of the outer view-ray loop is now load-bearing — uniform-bound loops can't be unrolled, so the break becomes a real per-wavefront exit and dense sub-Neptune atmospheres bail mid-march once view transmittance drops below 3%. The original far-camera heuristic from the optimisation plan was abandoned because the user's worst-case lag was at *close* zoom (where that rule wouldn't trigger); thickness-only is simpler and directly targets the bottleneck. Mars-like band kept the lower `jSteps = 3` even after the `iSteps` bump — inner-loop integrates a smooth exponential along the sun ray, tolerates fewer samples without producing visible banding.
- **Bake deferral + reveal animation hides the entry-frame stall and the textureless flash**: The remaining ~50 ms entry stall after step 2 was the bake itself (cloud + surface bakes still ran synchronously on the first frame after seed change). Step 3 adds two state flags on `PlanetRenderer`: `bakePending` and `hasBakeCompleted`. On a fresh seed: frame N records `bakePending = true` and skips the bake entirely; frame N+1 runs the bake (the GPU stalls for ~50 ms but `uReveal = 0`, so nothing's visible during the stall); frame N+2 onward ramps `uReveal` from 0 → 1 over 200 ms. The original plan called step 3 a defer-and-show-flat-planet trick where the user would see one untextured frame; combining it with a fade-in eliminates that artifact entirely — `uReveal = 0` on both the deferred frame and the bake frame means the textureless frame is never visible, and the planet appears to "light up from nothing" rather than pop in. User asked specifically about the lit-up-from-nothing concept and it turned out to be both nicer UX and a cleaner solution to the textureless-frame problem.
- **`uReveal` multiplied into HDR `col` before exposure / bloom, not post-tonemap**: Applying the multiply pre-tonemap means corona, spikes, and bloom all fade in together — bloom dims with the scene rather than blooming over a partially-faded planet. The fade curve is slightly non-linear in display space because Reinhard tonemap maps `0.5 * x → tonemap(0.5x)` rather than `0.5 * tonemap(x)`, but the result still reads cleanly as "fading from black to full" and the visual integration of all light-emitting effects (planet, atmosphere, corona, spikes, bloom) into one coherent reveal is worth the curve mismatch.
- **Reveal triggered only on `seed != lastSeedRequested`, not on every recomposition**: Tracking `lastSeedRequested` (separate from `lastBakedSeed` and `lastSurfaceSeed`) means intra-renderer planet swaps trigger the deferred bake + fade-in, but re-binding the same params (e.g., due to camera-state recomposition) doesn't restart the animation. Fresh AndroidView instantiation always triggers the fade-in because `lastSeedRequested` initialises to `Long.MIN_VALUE` — desired behaviour, since fresh GL state needs the bake regardless.
- **Compose-side `.background(Color.Black)` and `setBackgroundColor` on SurfaceView do NOT work — they hide the GL output entirely**: First attempt at hiding the dark-blue punch-hole exposure was to add `Modifier.background(Color.Black)` on `PlanetDetailContent`'s outer Box and `glView.setBackgroundColor(BLACK)` on the GLSurfaceView. Both broke the GL view: the Compose `.background()` paints opaque black at the SurfaceView's bounds in the window's surface, which Compose then treats as the final pixel — the SurfaceView's "punch-hole" mark gets overridden, the GL surface stays composited *behind* the now-opaque window pixels, and the planet becomes invisible. Same effective result with the View-level `setBackgroundColor`. Reverted both.
- **Theme-level `android:windowBackground` change alone wasn't enough — what shows through the punch-hole is whichever Compose ancestor painted last at the SurfaceView's bounds**: Second attempt was changing `<item name="android:windowBackground">` in `themes.xml` to `#000000`. Still didn't fix the dark-blue exposure during slide-in. Reason: the `windowBackground` only shows when Compose hasn't painted anything at that location. SystemScreen's outer Box has `.background(colors.background)` on a `Modifier.fillMaxSize()` that covers the entire System tab — that paint reaches the SurfaceView's bounds during page-slide animations (when the new page hasn't yet covered the slide-in region) and overrides the window background. Real fix: change SystemScreen's outer Box `.background(colors.background)` → `.background(Color.Black)`. Each child page (StarSearchContent, SystemDetailContent, PlanetDetailContent) now paints its own `colors.background` where needed, so the only effect is during transition periods — exactly when we want black showing. Theme `windowBackground` is also kept at `#000000` for cold-start splash consistency.
- **`StarSearchContent` got its own `.background(colors.background)` to compensate**: The search page never owned its own page-level background — it inherited from SystemScreen's outer Box. With that outer Box now black, the search page would have shown black between cards. Added `modifier.background(colors.background)` to its `LazyColumn`. SystemDetailContent and PlanetDetailContent already painted their own dark-blue backgrounds on their text-content sections, so they didn't need any change.
- **`cachedPlanetEntry` resolution made synchronous; bookmark icon and toggle no longer lag a composition cycle behind `selectedPlanetName`**: The LaunchedEffect-only update of `cachedPlanetEntry` lagged by a composition because it ran after composition completed, while `selectedPlanetName` updated synchronously. The bookmark icon read from `cachedPlanetEntry.planet.name` while `onBookmarkSelectedPlanetToggle` read from `selectedPlanetName.value`, producing two distinct user-visible bugs from the same root cause. (1) Same-system A→B navigation flashed the previous planet's bookmark state during the slide-in until the cache caught up. (2) For Catalog/StarMap → PLANET → top-left back to DETAIL → re-open a planet from DETAIL: the bookmark button's push animation fired but the icon never updated, because the icon's `isBookmarked` was being computed against the cached previous entry while the toggle operated on the current selectedPlanetName — the two always disagreed about which planet was being toggled. Fix: compute `resolvedPlanetEntry` synchronously in composition (prefer current state's match, fall back to cache when no match — preserves the cross-system continuity feature where systemDetail hasn't loaded the new system yet). The cache is kept as a fallback and updated reactively for the next composition. Both bugs resolved.
- **`WATER_ICE` and `NITROGEN_ICE` palettes swapped tones; frozen sea now picks same salt as polar cap**: User reported that on a given terrestrial planet the polar cap and the frozen sea were visibly different shades — both came from the `WATER_ICE` palette but with different salts (cap=4, sea=2), so `pickColor`'s deterministic-but-salted index selection gave different palette entries on the same seed. Two changes to fix: (a) in `PlanetGlobeParams.kt` and `AtmospherePresetScreen.kt` the frozen-sea `pickColor` now uses salt 4 — same as the cap — so they always pick the same index and the two ice surfaces match on any given planet. (b) The `WATER_ICE` palette was rewritten as neutral whites with only a faint cool cast (`0xFFF2F4F4`, `0xFFE8ECEC`, `0xFFFAFAFA`, `0xFFDDE0E0`) — closer to how Earth's and Mars's water-ice caps actually read. The bluer shades it used to hold (`0xFFE8F0F8`, `0xFFD0E0F0`, `0xFFF0F4F8`, `0xFFC0D4E8`) moved to `NITROGEN_ICE`, replacing its previous pale-pinkish set. The pinkish look came from a Pluto-by-association assumption, but Pluto's pink is tholin chemistry layered over the N2, not the N2 itself; pure N2 ice scatters short wavelengths and reads as cool/blue-white, so the new palette is also more physically defensible. Both palette changes cascade through every consumer (icons, settings preview, solar system profiles) so the visual identity stays consistent across surfaces.
- **Particulate-bias modifier on Rayleigh values for thin atmospheres**: Pure molecular Rayleigh scattering follows λ⁻⁴, which always produces a blue sky regardless of composition. But on real Mars-like to sub-Earth-like atmospheres the molecular scattering is barely visible at the actual densities, and the visible sky colour comes from suspended surface dust — Mars's butterscotch sky is iron-oxide dust scattering, not the thin CO₂ doing molecular Rayleigh. User reported that all thin atmospheres in the app trended blue. Fix in `AtmosphereOpticsDeriver.derive`: after the molecular Rayleigh values are computed, apply a particulate bias that sways them toward the surface-derived dust tint, with a **cubic** falloff over density: 1.0 at Mars-thin (≤0.05× Earth), ~0.40 at d=0.3, ~0.15 at d=0.5, ~0.03 at d=0.7, 0 at Earth (≥1.0). The cubic curve was chosen after testing a linear ramp gave too much tint in the sub-Earth middle range — Mars still wants full bias but a half-density planet wants only a gentle nudge. Total Rayleigh magnitude is preserved — only the spectral distribution shifts, so the atmosphere stays as visible as before, just dust-tinted. New `computeDustTint` helper derives the tint from `surface.iron / silicates / sulfur / carbon` using the same per-material RGB signatures as `dustFogColor`. Knowingly violates the "molecular base values" rule for thin atmospheres because that rule produces physically wrong sky colours when particulate scattering dominates molecular; thicker atmospheres (Earth and above) still get pure molecular Rayleigh because for them the rule produces the right answer.
- **Discrete bright spots (faculae) on the procedural star surface — two-layer structure for granular detail**: User wanted occasional bright patches that read as distinct features rather than just intensified granulation. First attempt was a single low-frequency noise + uniform tint, which produced smooth blob shapes. Switched to a two-layer structure so spots read as "concentrated areas of smaller granular cells" matching the dark intergranular lanes' visual character: (1) low-frequency mask (`noise3d` at scale 1.5, threshold `smoothstep(0.45, 0.65)`) defines WHERE the spots are — only the rare positive tail of the region noise produces visible patches, so coverage stays well under the dark lanes' surface area; (2) high-frequency cell layer (scale `uNoiseScale1 * 2` — about half the size of the surrounding granulation cells) provides the granular detail INSIDE each spot, clamped to positive contributions so cells only brighten within the mask. Spot tint at `uStarColor * 2.0`, additive boost at `× 3.5 × uStarIntensity`. Independent time drift on each layer: 0.04 on the mask (slow region evolution), 0.3 on the cells (visibly boiling small granulation inside). Sample point is the same `surfPt` used by the main granulation (already rotated by `uSpinAngle`) so spots rotate with the star surface. Brown-dwarf textured path is intentionally unaffected.

- **Transit-midpoint → mean-anomaly conversion now uses Kepler's equation with ω and e (was hard-coded `π/2`)**: User reported orbital configurations didn't match NASA's Exoplanet Archive. UTC-vs-TDB time-system drift is ~8 minutes max and negligible against the median exoplanet orbital period, so not worth correcting separately. - **`smoothToRealTime()` interpolates both planet and binary positions over 400 ms instead of snapping**: User reported the slider's RT step caused planets to pop from their fast-forward position to their real-time position. Added `smoothToRealTime()` alongside the existing `resetToRealTime()` (kept for the initial-load path where there's nothing on screen to interpolate from). The smooth path captures `interpStartSimJD = referenceJD + accumulatedDays` and `interpStartAccumulated = accumulatedDays` at the start, then each `onFrame` slides both: `interpolatedSimJD` Hermite-eases from `interpStartSimJD` toward the current real-time JD (epoch-anchored planet phases follow), and `accumulatedDays` eases from `interpStartAccumulated` down to 0 (binary stars follow back to M=0). `referenceJD` is reconstructed each frame as `interpolatedSimJD - accumulatedDays` so the planet target stays consistent while `accumulatedDays` is being driven down. Initial implementation only animated `referenceJD` and let binaries stay where they were — user pushed back wanting binaries to revert too, so the second axis was added. - **Terrestrial domain-warp strength fixed at 45 %, replacing the regional `dynWarp` modulation**: `terrestrial_bake.frag` previously varied the domain-warp strength by region via `dynWarp = 1.2 − sup * 0.9` (range 0.3–1.2, driven by the same suppressor noise that varies macro FBM scale). - **Camera-snapshot atomic read fixes one-frame "drag-direction flash" on every globe / map view**: User reported a one-frame jitter in the planet/star globe view that flashed in the direction of swiping — happened at any drag speed but never interrupted the motion. Root cause: `CameraController.viewMatrix`, `projectionMatrix`, and `eyePosition` were three independent volatile reads. The renderer's `onDrawFrame` read them sequentially while `tickDamping` (Choreographer, UI thread) wrote `eyePosition` and then swapped `matrices` on every vsync during active drag. If a tick fired *between* the renderer's reads, the GL frame paired a pre-tick `viewMatrix` with a post-tick `eyePosition` — the ray-march origin (`uCameraPos`) was one tick ahead of the screen-space mapping (`uInverseView`), producing a one-frame visual offset matching the camera's angular velocity. Next frame's reads were consistent → smooth motion resumed. **Fix**: bundled `eye` into `MatrixPair` so view + projection + eye swap atomically as one volatile assignment, and added `CameraController.snapshot()` returning a `Snapshot(view, projection, eye)` from a single read of `matrices`. `PlanetRenderer`, `StarRenderer`, `StarMapRenderer`, and `OrbitalRenderer` now snapshot once at the top of their frame and reuse the captured fields for every matrix read in that frame — including passing `cam.view` down to `OrbitalRenderer.rebuildPlanetPositions` and `StarMapRenderer.updateRings` so their internal matrix uses match the frame's MVP. The legacy single-field getters (`viewMatrix`, `projectionMatrix`, `eyePosition`) are kept for tap-projection callers that only read one field; they all now read from the same `matrices` reference so a single getter call is internally consistent (only cross-call reads tore, which only the renderers did).

- **Star disc soft edge + outer saturation blowout**: The disc's geometric edge fade widened from a 10% smoothstep to 15% so the limb reads as a soft overexposed boundary rather than a hard circle (the existing limb darkening and near-white saturated tint conventions continue to apply). Same `computeSunOcclusion × computeScreenEdgeFade × dimFade × pow(sunOcc, 2.5)` gating as the spikes so it fades coherently during eclipse / off-screen / sub-threshold flux. **Extent**: exponential decay with characteristic distance `drawRadius × 0.6` — tightly hugs the disc, no hard-coded angular constant. The upper bound landed at 0.8 after testing — a briefer 0.7 threshold (used during the first attempt at this formulation) cut TRAPPIST-1 h to ~20 %, which still read as "the spikes are gone" against the dim M-dwarf glare.

Real rings (and SpaceEngine-style renders) noticeably brighten when seen near edge-on because the line of sight passes through more material per pixel — for an optically thin ring, brightness scales as `1/sin(B_view)`. Replaced the cosine term with `ringPhotometric(sunDir, viewDir, ringDens)` in [planet.frag](app/src/main/assets/shaders/planet.frag), implementing the Cuzzi/Cassini-team form `I/F ∝ (μ₀ / (μ₀ + μ)) × (1 − exp(−τ × (1/μ₀ + 1/μ)))`. The path-length factor `(1/μ₀ + 1/μ)` is what makes the ring brighten near edge-on viewing — as μ drops, the integrated optical depth grows and the exponential saturates toward the prefactor, which is the same Lommel-Seeliger `μ₀ / (μ₀ + μ)` term we use for airless surfaces. Optically thick portions saturate to the prefactor; thin sections (`ringDens < 0.3`) show dramatic shallow-angle brightening. `tau = ringDens × 1.5` maps a dense `ringDens = 1.0` to Saturn-B-ring τ ≈ 1.5; thin Uranus-style `ringDens ≈ 0.1` lands at τ ≈ 0.15. μ₀ and μ floored at 0.05 to keep the 1/μ term finite at exact edge-on. Applied at all six ring-lighting call sites (airless surface ring-in-front × 2, atmosphere surface ring-in-front × 1, ring-in-empty-space × 1, ring-behind-atmosphere × 1, ring-in-front-of-atmosphere × 1) by replacing the inline `ringNdotL` computation with a `ringPhoto` from the shared helper.
- **Opposition surge added to the airless-rocky lighting path**: Lommel-Seeliger alone gives a flat disc but doesn't reproduce the sharp brightness spike a regolith body shows near phase angle 0 (observer between sun and planet — the "full Moon" configuration). Real regolith has a coherent-back-scatter peak from shadow-hiding and constructive interference near anti-illumination geometry, adding ~25–60 % brightness at exact opposition over a narrow ~5° FWHM. Added `oppositionSurge(sunDir, viewDir)` helper next to `hapkeDiffuse` in [planet.frag](app/src/main/assets/shaders/planet.frag): early-outs to 1.0 when `uHapkeStrength <= 0` (atmospheric world — no regolith physics applies), otherwise returns `1 + uHapkeStrength * 0.5 * exp(-alpha / 0.10)` where `alpha = acos(dot(sunDir, viewDir))`. Peak amplitude 0.5 means a full-strength Hapke body brightens 50 % at exact opposition; width 0.10 rad (~5.7°) so the surge is invisible past mid-quarter phase, matching the lunar opposition-effect FWHM. Both the amplitude and the width's effective gating scale with `uHapkeStrength` so the surge fades in alongside the Hapke diffuse term rather than appearing as a discontinuous boost on planets that don't otherwise show regolith physics. Applied in the airless surface path to both primary and secondary star contributions; specular still uses raw `nDotL` (the Phong term is geometric and the surge is a regolith-physics term).
- **Chevron-banding threshold raised from 6.0 to 7.5 M⊕**: Original threshold left a population of low-mass sub-Neptunes just past the cutoff that visually felt like they should match Venus's super-rotation regime — the boundary between "still small enough for chevron flow" and "ice-giant zonal jets dominate" sits closer to ~7–8 M⊕ in the relevant atmospheric depth / rotation rate / temperature space than to 6. Threshold bump in `applyChevron` of [GasGiantGenerator.kt](domain/src/main/java/com/tadmor/domain/classification/visual/GasGiantGenerator.kt) catches that population without spilling into Neptune-mass territory.
- **Hapke / Lommel-Seeliger lighting for airless rocky bodies**: Default Lambertian shading paints a smooth brightness fall-off from subsolar point to terminator, which reads correctly for atmospheric worlds but wrong for the Moon — a real lunar disc looks almost uniformly bright out to the terminator because regolith back-scatters strongly toward the light source. **Implementation**: new `uHapkeStrength` uniform on `planet.frag` plus a `hapkeDiffuse(nDotL, nDotV)` helper that returns `mix(nDotL, 2 × nDotL / (nDotL + nDotV), uHapkeStrength)`. The `2×` factor keeps the subsolar brightness at the Lambert level so the strength dial blends cleanly. Wired into the airless surface path only — both primary and secondary star contributions in the `uAtmosphereThickness <= 0` branch read `diffuse = hapkeDiffuse(...)` instead of the raw `nDotL`. Surface specular still uses raw `nDotL` since the Phong term is geometric, not a regolith-physics term. **CPU gate** (in `buildPlanetGlobeParams`): only activates when ALL of (a) terrestrial bake present, (b) `optics.atmosphereThicknessKm <= 0`, (c) `surfaceTemperatureK < 900` (molten worlds shouldn't read flat — their warm-pixel emission needs the Lambert cue to ground the limb), (d) `water + methane + tholins < 0.25` (Lommel-Seeliger physics describes loose regolith, not ice sheets or hydrocarbon seas), (e) `craterProfile.density >= 0.25` (crater density is the strongest single signal that a surface is unweathered powder — Mercury, Moon, Callisto-class). Strength scales with crater density past the threshold so heavily-cratered worlds reach full Hapke and moderately-cratered ones blend smoothly into Lambertian.
- **L-dwarf surfaces switch from FBM granulation to Voronoi cell pattern**: L and M dwarfs occupy adjacent slices of the H-R diagram and share the same deep-red colour temperature, so they were rendering nearly identically with the standard FBM-warped granulation pattern used for cool stars. Real L dwarfs have very different surface phenomenology — extremely dense silicate clouds and a complex multi-layer optical depth that reads more like cracked basalt / regolith plates than the smooth convection cells of a hotter star. **Implementation**: new `uIsLDwarf` uniform on `star_globe.frag`, set by `StarRenderer` from a new `isLDwarf: Boolean` on `StarGlobeParams`. Detection in `buildStarGlobeParams` checks for explicit "L" spectral prefix OR (no spectral type AND Teff in 1300–2400 K). When set, the surface branch swaps the FBM + supergranulation + ridged-noise path for a 2-octave 3D Voronoi pattern — `(F2 − F1)` cell-edge cracks for dark intergranular lanes, `1 − smoothstep(0, 0.45, F1)` interior brightness for hot "upwelling" cell centres, no domain warp on the input so cells stay recognisably polygonal rather than smearing into FBM-like blobs. Hot/cool tint is similar to the FBM path (`uStarColor × 1.5` vs `× (0.45, 0.30, 0.20)`) so the colour signature matches — the topology is what changes. Emissive boost on cell interiors using the `(1 − edge)` mask mirrors the FBM path's hot-granulation pop. Outer fresnel edge brightening and corona stages remain shared — only the convection signal itself diverges between the two paths. **Followup — domain warp curves the cell boundaries**: pure Voronoi gives dead-straight perpendicular-bisector edges between cells, which reads as artificial polygonal mesh rather than the curving cracked-basalt look the reference imagery shows. Added a low-frequency FBM domain warp to the Voronoi input (warp magnitude ≈ 0.30 of one cell width, FBM frequency at 0.6× the cell frequency) so the lattice deforms — adjacent cells curve coherently rather than jittering individually. Magnitude is calibrated to bend the cell boundaries visibly without smearing cells into FBM-like blobs that would lose the polygonal identity Voronoi gives us. Sampled at the warped position for both octaves so the two scales remain visually coherent.
- **Brown-dwarf atmosphere height shortened: Teff-driven scale-height proxy replaces 5 % constant**: The L/T/Y brown-dwarf limb halo was using a fixed 5 % R★ extent for every dwarf regardless of temperature, after an earlier pass dropped it from 15 %. Physical reality is that atmospheric scale height `H = kT/(mg)` couples both temperature and gravity, and brown dwarfs sit on the electron-degeneracy plateau where radius is essentially fixed at ~1 R_jupiter regardless of mass — so `H/R` scales roughly as `T/g`, and gravity in turn scales with mass (`M/R²` with R fixed). Most catalogued BDs are missing M / L / radius entries, so a Teff-driven proxy is the only practical path. Cooler dwarfs are typically lower mass (older and closer to the planetary boundary), so the `T/g` ratio — and thus `H/R` — stays roughly conserved across the BD range at ≈ 6×10⁻⁵. Visible Rayleigh/Mie halo extends ~6–10 scale heights, putting the apparent limb extent in star radii at ~1 % near the L/T boundary. **Final formula**: `atmLimbExtent = (teff × 0.000012).coerceIn(0.005, 0.020)` — Y dwarfs (~400 K) get a 0.005 R★ hairline, T dwarfs at the 1300 K upper boundary get 0.016 R★, fraction of the earlier 0.05 prototype. Matches the actual physical extent of a brown-dwarf atmosphere far better than the constant. Banded gas-giant texture path and per-band limb-colour selection from the same earlier work are unchanged — only the *radial extent* of the halo varies with Teff now.
- **Two-sun planet globe for circumbinary planets**: Circumbinary planets — Kepler-16 b, TOI-1338 b, the rest of the ~30 catalogued binaries — now render BOTH stars in the planet detail globe view at their time-averaged static angular separation around the barycenter direction. **Placement math**: the existing single-sun direction (axial-tilt + season-phase computed in `buildPlanetGlobeParams`) now represents the *barycenter* direction. The instantaneous max angular separation is `binaryStarSeparationAU / planetSmaAU` (small-angle valid for every catalogued circumbinary since `a_b << a_p`), reached when the binary axis is perpendicular to the line of sight. Since every transit-discovered circumbinary is viewed near edge-on, the projected separation oscillates as `a_b × |sin(φ)|` over the orbital phase, and `⟨|sin(φ)|⟩ = 2/π`. So we use `θ = (2/π) × a_b / a_p` for a representative "what you'd usually see" pose rather than the extreme. For Kepler-16 (`a_b ≈ 0.22 AU`, `a_p ≈ 0.7 AU`) that's ≈ 11.5° — still reads as two distinct suns without being cartoonishly wide. Splits symmetrically in azimuth: primary at `(sunAz − θ/2)`, secondary at `(sunAz + θ/2)`. Static placement — per user direction, since the planet's distance from the barycenter is itself static (we don't animate planet position on the globe page), the stars' positions follow the same convention. Real binary orbital motion (days to weeks) is a future enhancement. **Shader pipeline (planet.frag)**: every per-sun contribution is duplicated behind a `uHasSecondarySun` uniform branch: atmosphere ray-march in-scattering accumulates a separate `totalScattering2` integrating phase functions / sun-transmittance for the second star, surface Lambertian sums per-sun nDotL × atmosphere atten × ring shadow, sun disc + corona + diffraction spikes paint twice in `writeFinal`, ring shadows on the surface multiply per-sun (one star's ring band can occlude that star's lighting without affecting the other), cloud-deck Lambertian sums per-sun bumped diffuse × per-sun atmosphere transmittance above clouds, and surface cloud shadows project the cloud noise along each sun's direction. Helpers (`getRingShadow`, `ringInPlanetShadow`, `computeSunOcclusion`, `computeScreenEdgeFade`, `computeStarSpikes`, `computeStarCorona`, `surfaceSpecular`) all take sun parameters explicitly rather than reading the primary uniform — same helpers serve both stars. **Cost**: the atmosphere ray-march inner loop runs the J-step sun-transmittance integration twice (the dominant per-step cost), so circumbinary planets pay roughly 2× atmosphere-loop work. Only ~30 of 6,000+ planets are circumbinary, so overall perf is unchanged; the `uHasSecondarySun == 0` branch carries a single uniform check and zero fragment-level overhead for the other 99.5%. **Brightness calibration**: auto-exposure switches to `20 / (sunIntensity + sun2Intensity)` so a binary's combined flux doesn't blow out the Reinhard tonemap. Each star contributes its own luminosity-derived intensity (`L_i / a_p²` per star, same `× 40` scale factor as single-star path) and its own blackbody-derived colour, so a K-dwarf + M-dwarf pair shows two distinct tints in the sky. Both stars at the same `sunDistanceAU = a_p` since the binary separation is negligible compared to the planet's orbital radius. **Data wiring**: `buildPlanetGlobeParams` gains optional `companionStar: Star?` + `binaryStarSeparationAU: Double = 0.0` parameters; SystemScreen passes them from `renderDetail.companionStars.firstOrNull()` + `resolvedOrbitalState?.binaryStarSeparationAU` when `entry.planet.cbFlag` is set. If either is missing (system or orbital state hasn't loaded yet on a fresh navigation) the builder falls back to a single-star render and the next composition upgrades to two-sun once the data arrives. **Tidal-lock substellar still locks to the primary** (`uSunDirection`) — the bullseye cloud pattern is gated on `CloudType == WATER` upstream, and circumbinary tidal locking targets the barycenter rather than either individual star, so the primary direction stays a reasonable proxy for the substellar pole. Future refinement could use the brighter-star direction if the disparity in intensity is large.
- **Catalog hardened against corrupt-name DTOs (4-layer defense)**: A duplicate `TOI-663 b` row arrived from the TAP catalogue with `pl_name` set to a 266 003-character run of garbage bytes (rendered as `?` in logcat — likely binary stored as UTF-8). Symptoms in the field: 20 + s freeze with ~400 k major page faults during catalog name-sort (TimSort allocated a fresh `String.lowercase()` copy of the bad name on every pairwise comparison → gigabytes of churn → swap thrash) and an "empty card that expands infinitely" when the corrupted row sorted to the top (the BasicText layout buffer for that string grew the row's height without bound, leaving only the icon visible above the fold). Fixed in four places so the defenses don't depend on which entry point sees the bad data first:
  - **Sort comparator** in [CatalogViewModel.kt](app/src/main/java/com/tadmor/app/ui/catalog/CatalogViewModel.kt) precomputes lowercased keys once per entry (with a 256-char sort-key cap and a warning log) instead of allocating per comparison — bounds total allocation at O(n × name length).
  - **Display string** in [CatalogScreen.kt](app/src/main/java/com/tadmor/app/ui/catalog/CatalogScreen.kt) caps the rendered name at 80 chars + "…" so the card stays sensibly sized even if a long name somehow slips through.
  - **Entity → domain mapper** in [EntityToDomainMapper.kt](data/src/main/java/com/tadmor/data/mapper/EntityToDomainMapper.kt) sanitises every `Planet.name` to ≤ 64 chars on read, so classifier, visual pipeline, system-detail UI, *and* catalog UI all see safe strings regardless of caller — the freeze on opening the planet's system page went away with this single boundary cap.
  - **Ingestion filter + DB cleanup** in [PlanetRepositoryImpl.kt](data/src/main/java/com/tadmor/data/repository/PlanetRepositoryImpl.kt) drops DTOs with `plName.length > 128` before they reach `upsertAll`, AND calls a new `PlanetDao.deleteRowsWithCorruptName(maxLen)` at the start of every sync transaction so existing bad rows from prior syncs get evicted on the next sync rather than waiting to be overwritten organically.

  Each layer logs a Timber warning with the offending hostname + first 40 chars of the bad name, so future corruption events are immediately traceable. NASA TAP designations cap at ~30 chars in practice; the 64/128/256 char tiers leave plenty of headroom for legitimate long names (BD/HD/HIP variants with extra suffixes) while everything past that is treated as data corruption.
